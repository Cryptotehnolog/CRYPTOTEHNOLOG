# Dual Control (Operator Gate) для критических операций

**Дата:** 2026-03-09  
**Статус:** Принято  

## Контекст
Фаза 2 определила Operator Gate как механизм защиты от ошибочных или злонамеренных действий.

- **Фаза:** 2 — Control Plane
- **Компонент:** src/cryptotechnolog/core/operator_gate.py
- **Класс стратегии:** SFT (безопасность)

### Почему Dual Control:
1. **Защита от ошибок** — один оператор может случайно остановить систему
2. **Защита от злоумышленника** — если аккаунт одного оператора скомпрометирован
3. **Аудит** — требуется два подтверждения для критических действий

## Рассмотренные альтернативы
1. **Single operator** — риск ошибки/злоупотребления
2. **Тройное подтверждение** — избыточно для большинства случаев
3. **Biometric auth** — сложность реализации, нет необходимости

## Решение
Выбрана **dual-control модель**:

### Критические операции (требуют 2 подтверждений):
- → HALT (полная остановка)
- → RECOVERY (восстановление)
- Изменение лимитов риска
- Изменение конфигурации

### Некритические операции (одно подтверждение):
- → TRADING (начало торговли)
- → DEGRADED (деградация)
- Запрос метрик

### Поведение:
```
1. Оператор 1 запрашивает переход → HALT
2. Operator Gate создаёт pending request с timeout 5 минут
3. Оператор 2 подтверждает или отклоняет
4. При подтверждении → переход выполняется
5. При timeout → запрос отклоняется
```

## Последствия
### Плюсы:
- **Безопасность** — защита от случайных/злонамеренных действий
- **Аудит** — все запросы логируются
- **Соответствие** — соответствие регуляторным требованиям

### Минусы:
- **Задержка** — 2 подтверждения занимают время
- **SPOF** — если один оператор недоступен
- **Сложность** — UI для подтверждения
