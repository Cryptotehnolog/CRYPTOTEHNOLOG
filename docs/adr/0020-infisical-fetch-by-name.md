# Infisical: получение секретов по одному без права listSecrets

**Дата:** 2026-03-17  
**Статус:** Принято  

## Контекст
- **Фаза/компонент:** Phase 0 - Управление конфигурацией и секретами
- **Класс стратегии:** Не применимо (инфраструктурный код)
- **Проблема:** Infisical Machine Identity не имеет права `listSecrets` для получения списка всех секретов в папке через API `/api/v3/secrets`. При попытке получить список возвращается пустой массив.

## Рассмотренные альтернативы
1. **Получение списка через `/api/v3/secrets`** - требует права listSecrets, которого нет в Machine Identity
2. **Хардкод списка секретов в коде** - требует изменения кода при добавлении новых секретов
3. **Получение секретов по одному через `/api/v3/secrets/raw/{name}`** - работает без listSecrets, имена задаются через конфигурацию

## Решение
Выбрана третья альтернатива - получение секретов по одному с именами, заданными через переменную окружения.

### Реализация:
- **Конфигурация:** `.env.infisical`
  ```bash
  INFISICAL_SECRET_PATHS=/staging/crypto,/production/crypto,/staging/telegram,...
  INFISICAL_SECRET_KEYS=BINANCE_API_KEY,BINANCE_API_SECRET,...
  ```
- **Код:** [`src/cryptotechnolog/config/providers.py`](../../src/cryptotechnolog/config/providers.py)
  - `_get_secret_keys()` - читает ключи из переменной окружения
  - `_fetch_single_secret()` - получает каждый секрет отдельно по имени

## Последствия
- **Плюсы:**
  - Не требует изменения кода при добавлении новых секретов
  - Работает с текущими правами Machine Identity (readValue + describeSecret)
  - Гибкая настройка через переменные окружения
- **Минусы:**
  - Нужно явно указывать имена всех секретов в конфигурации
  - Больше HTTP-запросов (по одному на каждый ключ)

## Связанные ADR
- Связано с ADR-0017 (Config Manager Architecture) - часть системы управления конфигурацией
