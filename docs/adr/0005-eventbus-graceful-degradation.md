# Graceful Degradation: возврат ошибки при переполнении Event Bus

**Дата:** 2026-03-09  
**Статус:** Принято  

## Контекст
Фаза 1 определила поведение Event Bus при переполнении буфера — возврат ошибки вместо блокировки.

- **Фаза:** 1 — Ядро инфраструктуры
- **Компонент:** crates/eventbus (все backend)
- **Класс стратегии:** HFT (предотвращение deadlocks)

### Почему не блокировка:
При высокой нагрузке блокировкаpublisher может привести к:
1. **Deadlock** — вся система останавливается
2. **Priority inversion** — low-priority события блокируют critical
3. **Head-of-line blocking** — один медленный подписчик блокирует всех

## Рассмотренные альтернативы
1. **Блокировка (blocking)** — классический подход, но риск deadlock
2. **Drop oldest (drop tail)** — потеря новых событий
3. **Drop specific (drop priority)** — удалять low-priority при перегрузке

## Решение
Выбрана **graceful degradation** — возврат ошибки:

```rust
// При переполнении буфера возвращаем ошибку
pub enum EventBusError {
    BufferFull {
        capacity: usize,
        current_size: usize,
        priority: Priority,
    },
    // ...
}

// Вызывающий код решает что делать:
// 1. Повторить попытку
// 2. Записать в fallback storage
// 3. Залогировать и продолжить
// 4. Критическая ошибка -> shutdown
```

### Поведение:
- **Bounded capacity** — предотвращает неконтролируемый рост памяти
- **Возврат error** — не блокирует publisher
- **Приоритизация** — при переполнении сохраняем high-priority

## Последствия
### Плюсы:
- **Нет deadlock** — система продолжает работать
- **Предсказуемость** — известное поведение при перегрузке
- **Управление** — приложение решает как реагировать
- **Защита от перегрузки** — bounded memory

### Минусы:
- **Потеря событий** — нужно обрабатывать error case
- **Сложность** — publisher должен обрабатывать ошибки
- **Мониторинг** — нужно отслеживать ошибки переполнения
