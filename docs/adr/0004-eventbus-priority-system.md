# Система приоритетов событий в Event Bus

**Дата:** 2026-03-09  
**Статус:** Принято  

## Контекст
Фаза 1 определила 4 уровня приоритета событий для обеспечения дифференцированного обслуживания.

- **Фаза:** 1 — Ядро инфраструктуры
- **Компонент:** crates/eventbus (priority.rs)
- **Класс стратегии:** HFT (критические торговые сигналы)

### Уровни приоритета:
```rust
pub enum Priority {
    Critical = 0,  // Kill switches, экстренные остановы
    High = 1,      // Risk violations, критические уведомления
    Normal = 2,    // Обычные события торговли
    Low = 3,       // Метрики, логи, аудит
}
```

## Рассмотренные альтернативы
1. **Binary приоритет** (Critical/Normal) — слишком упрощённо
2. **8 уровней приоритета** — избыточно для текущих сценариев
3. **Динамические приоритеты** — усложнение, нет необходимости

## Решение
Выбрана **4-уровневая система приоритетов**:

### Примеры использования:
| Приоритет | События | Требования доставки |
|-----------|---------|---------------------|
| Critical | KILL_SWITCH_TRIGGERED, EMERGENCY_SHUTDOWN | sync, CRITICAL priority |
| High | RISK_VIOLATION, ORDER_REJECTED | async, HIGH priority |
| Normal | ORDER_FILLED, POSITION_OPENED | async, NORMAL priority |
| Low | METRICS_PUBLISHED, HEALTH_CHECK | best-effort, LOW priority |

### Гарантии:
- **Critical:** синхронная обработка (блокировка отправителя)
- **High:** at-least-once delivery
- **Normal:** at-most-once delivery
- **Low:** best-effort

## Последствия
### Плюсы:
- **Предсказуемость:** Critical события всегда обрабатываются первыми
- **Защита от перегрузки:** Low события могут быть отброшены при перегрузке
- **Соответствие требованиям:** разные компоненты имеют разные SLA

### Минусы:
- **Сложность реализации:** требуется priority queue
- **Голодание:** Low приоритет может долго ждать при высокой нагрузке
- **Конфигурация:** нужно правильно классифицировать события
