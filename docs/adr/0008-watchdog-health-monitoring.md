# Watchdog: Health Monitoring и Auto-Recovery

**Дата:** 2026-03-09  
**Статус:** Принято  

## Контекст
Фаза 2 определила Watchdog как компонент для мониторинга здоровья системы и автоматического восстановления.

- **Фаза:** 2 — Control Plane
- **Компонент:** src/cryptotechnolog/core/watchdog.py
- **Класс стратегии:** SFT (отказоустойчивость)

### Функции Watchdog:
1. **Health Checks** — периодическая проверка компонентов
2. **Auto-Recovery** — автоматическое восстановление при сбоях
3. **Escalation** — эскалация при невозможности восстановления
4. **Debouncing** — игнорирование единичных сбоев

## Рассмотренные альтернативы
1. **Только Manual Recovery** — слишком медленно для HFT
2. **Kubernetes Health Checks** — недостаточная бизнес-логика
3. **Only alerting** — нет автоматического восстановления

## Решение
Выбрана **активная модель** с автоматическим восстановлением:

### Health Checks:
```python
# Пример проверки
{
    "component": "execution_layer",
    "checks": [
        {"name": "api_latency", "threshold": 500, "unit": "ms"},
        {"name": "order_fill_rate", "threshold": 0.90},
    ]
}
```

### Auto-Recovery стратегии:
| Проблема | Действие |
|----------|----------|
| Redis disconnect | Переподключение |
| High latency | Перезапуск компонента |
| Component fail | Переход в DEGRADED |

### Debouncing:
- Единичные сбои игнорируются
- 3 последовательных сбоя → действие

## Последствия
### Плюсы:
- **Автономность** — система может восстанавливаться без человека
- **Быстрота** — минимизация downtime
- **Предсказуемость** — известное поведение при сбоях

### Минусы:
- **Сложность** — нужно правильно настроить thresholds
- **Risk** — auto-recovery может усугубить проблему
- **Logging** — много событий при проблемах
