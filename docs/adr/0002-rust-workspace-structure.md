# Структура Rust Workspace в crates/

**Дата:** 2026-03-09  
**Статус:** Принято  

## Контекст
Фаза 0 определила организацию Rust-кода как единый workspace с отдельными крейтами.

- **Фаза:** 0 — Подготовка среды
- **Компонент:** crates/ (Rust workspace)
- **Класс стратегии:** HFT (критические компоненты)

### Требования к структуре:
1. Изоляция компонентов для независимой разработки
2. Общие зависимости и воспроизводимые сборки
3. FFI для связи с Python

## Рассмотренные альтернативы
1. **Отдельные репозитории для каждого крейта** — сложность синхронизации версий
2. **Один большой крейт** — нарушение принципа единственной ответственности
3. **Подмодули git** — сложность CI/CD

## Решение
Выбрана **монорепо структура** с единым Cargo workspace:

### 6 крейтов в crates/:
1. **eventbus/** — высокопроизводительная шина событий
2. **risk-ledger/** — двойная бухгалтерия рисков
3. **audit-chain/** — криптографическая цепочка аудита
4. **execution-core/** — низкоуровневое исполнение ордеров
5. **ffi/** — Python привязки (PyO3)
6. **common/** — общие типы и утилиты

### Конфигурация Cargo.toml (workspace):
```toml
[workspace]
members = [
    "crates/eventbus",
    "crates/risk-ledger", 
    "crates/audit-chain",
    "crates/execution-core",
    "crates/ffi",
    "crates/common",
]
resolver = "2"
```

## Последствия
### Плюсы:
- **Общий Cargo.lock** — воспроизводимые сборки
- **Локальные зависимости** — крейты могут зависеть друг от друга
- **Единый CI/CD** — один pipeline для всех крейтов
- **Разделение ответственности** — каждый крейт отвечает за свою область

### Минусы:
- **Связанность** — изменение common требует пересборки всех крейтов
- **Размер workspace** — может расти со временем
