# Rate Limiting для Event Bus

**Дата:** 2026-03-09  
**Статус:** Принято  

## Контекст
Фаза 3 добавила rate limiting для защиты от перегрузки подписчиков.

- **Фаза:** 3 — Event Bus Enhancement
- **Компонент:** crates/eventbus/rate_limiter.rs
- **Класс стратегии:** HFT (защита от перегрузки)

### Зачем rate limiting:
1. **Защита подписчиков** — не захлебываются от событий
2. **Предсказуемость** — известная максимальная нагрузка
3. **Graceful degradation** — постепенное снижение нагрузки

## Рассмотренные альтернативы
1. **Token bucket** — сложнее в реализации
2. **Fixed window** — проще, но可能有 burst
3. **No rate limiting** — риск перегрузки

## Решение
Выбран **Sliding Window** с токенами:

### Алгоритм:
```rust
pub struct RateLimiter {
    tokens: f64,
    max_tokens: f64,
    refill_rate: f64,  // tokens per second
    last_refill: Instant,
}
```

### Конфигурация по умолчанию:
| Приоритет | Rate | Burst |
|-----------|------|-------|
| Critical | unlimited | unlimited |
| High | 1000/sec | 2000 |
| Normal | 500/sec | 1000 |
| Low | 100/sec | 200 |

### Поведение:
1. При publish → проверяем tokens
2. Если enough tokens → пропускаем, списываем
3. Если нет → Return Error
4. Tokens refill по rate

## Последствия
### Плюсы:
- **Защита** — подписчики не перегружаются
- **Контроль** — настраиваемые лимиты
- **Предсказуемость** — известная max нагрузка

### Минусы:
- **Потеря событий** — при превышении лимита
- **Сложность** — нужно правильно настроить
- **Burst handling** — нужно правильно обрабатывать
