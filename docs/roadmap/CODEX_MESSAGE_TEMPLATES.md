# CODEX MESSAGE TEMPLATES
## Короткие шаблоны рабочих сообщений для Codex в VS Code

---

## Назначение

Этот документ нужен, чтобы не формулировать каждый раз рабочие сообщения с нуля
и не терять phase discipline.

Шаблоны ниже не заменяют фазовый план.
Они помогают быстро запускать правильный режим работы.

---

## 1. Открытие новой фазы

Использовать, когда фаза ещё не начала реализовываться.

```text
Переходим к подготовке новой фазы.

Работаем в ветке: <branch>

Критически важно:
- весь ответ только на русском языке
- authoritative truth:
  - README.md
  - prompts/plan/<P_X>.md
  - relevant ADR
  - фактический код
- historical/reference prompts использовать только как reference
- не делать кодовые изменения до alignment summary

Перед любыми изменениями:
1. проверь/создай ветку от master
2. изучи README.md, prompts/plan/<P_X>.md, relevant ADR, фактический код
3. historical prompts используй только как reference

Что нужно сейчас:
- дать короткий phase-alignment summary

Что хочу получить:
1. как ты понимаешь core scope фазы
2. какие 1-2 первых безопасных шага реализации ты видишь
3. где historical prompts конфликтуют с текущей truth
4. какие architectural traps есть

Ничего не меняй в коде на этом шаге.
```

---

## 2. Узкий implementation-step

Использовать, когда scope шага уже зафиксирован.

```text
Продолжаем <P_X>.

Открываем следующий практический шаг:
<step name>

Работаем в ветке:
<branch>

Критически важно:
- весь ответ только на русском языке
- не выходить за scope текущей фазы
- не трогать postоронние dirty/untracked файлы
- не реализовывать явно вынесенные future lines

Текущая задача:
<короткая цель шага>

Что нужно сделать:
1. <пункт 1>
2. <пункт 2>
3. <пункт 3>

Что НЕ делать:
- <запрет 1>
- <запрет 2>
- <запрет 3>

Формат работы:
1. сначала коротко напиши, какие файлы планируешь менять и как понимаешь шаг
2. потом внеси изменения
3. после этого покажи результат, проверки и остаточные риски
```

## 3. Review / Audit

Использовать, когда нужно сначала получить findings, а не новый код.

```text
Сейчас не пишем новый код.
Нужен честный review / audit.

Критически важно:
- весь ответ только на русском языке
- это review-задача, не implementation-задача

Проверь:
- соответствие кода текущему phase plan
- наличие scope inflation
- runtime truth / health / diagnostics truth
- verification gaps

Формат ответа:
1. Замечания
2. Остаточные риски
3. Короткий вывод:
   - можно продолжать
   - или сначала нужен corrective step

Ничего не меняй в коде, если не найдёшь что-то критическое.
```

---

## 4. Doc / Release Truth Normalization

Использовать, когда код уже есть, но docs ещё отстают.

```text
Кодовую реализацию не расширяем.
Сейчас нужен только шаг:
Release / Documentation Truth Normalization

Нужно:
- синхронизировать README.md
- синхронизировать prompts/plan/<P_X>.md
- подготовить или обновить prompts/plan/<P_X>_RESULT.md

Критически важно:
- не добавлять новые features
- не менять архитектурную truth
- не расширять scope

После изменений покажи:
- какие документы изменены
- как теперь формулируется closure truth
- остались ли blockers перед финализацией
```

---

## 5. Финализация фазы

Использовать только после зелёного review и verification.

```text
Переходим к формальной финализации <P_X>.

Финализируем фазу как <version>.

Сначала:
1. покажи краткий git status
2. проверь, нет ли посторонних изменений
3. не включай в релиз parallel tracks и случайные файлы

После этого:
1. прогони финальные проверки
2. сделай релизный commit
3. создай tag
4. выполни merge
5. выполни push

Критически важно:
- ответ только на русском языке
- не делать destructive git-операций
- не подмешивать dashboard или другие parallel tracks
- если есть проблема на любом шаге, не исправлять молча
```

---

## Как использовать шаблоны

Шаблон сам по себе не является truth.

Перед его использованием всегда подставляются:

- текущая фаза;
- текущая ветка;
- current authoritative documents;
- конкретный scope текущего шага.

То есть шаблон — это только форма, а содержание всегда задаётся из phase truth.
