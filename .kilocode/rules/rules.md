# Правила проекта CRYPTOTEHNOLOG

## 1. Основные принципы и коммуникация

*   **Язык общения:** Все ответы, объяснения, комментарии к коду и запросы должны быть на **русском языке**, если явно не указано иное.
*   **Формат ответов:** При изменениях в коде показывайте только изменённые фрагменты (диффы), если файл не очень мал. Сложные архитектурные решения сначала кратко объясняйте, затем показывайте код.
*   **Роли:** При использовании Kilo Code выбирайте подходящую роль (Architect, Code, Ask) в зависимости от задачи.

## 2. Работа с кодом на Python

*   **Стиль кода (следуйте `pyproject.toml`):**
    *   **Линтер:** Используйте **Ruff**. Запускайте `ruff check src/ tests/` перед коммитом.
    *   **Форматтер:** Используйте **Black** с настройками по умолчанию (`black src/ tests/`).
    *   **Импорты:** Сортируйте с помощью **isort** (интегрирован в Ruff).
    *   **Типизация:** Обязательно используйте аннотации типов. Проверяйте типы через `mypy src/`. Для сложных структур (Polars, AutoTrader) применяйте `TYPE_CHECKING` для избежания циклических импортов.
*   **Структура модулей:** Следуйте существующей структуре в `src/`:
    *   `config/` — управление конфигурацией
    *   `core/` — базовые утилиты
    *   `risk/` — движок управления рисками
    *   `analysis/` — анализ рынка
    *   `execution/` — исполнение ордеров
    *   `models/` — модели данных
    *   `intelligence/` — интеллектуальный слой
    *   `strategy/` — торговые стратегии
    *   `observability/` — мониторинг и метрики
    Не создавайте новые корневые папки без согласования.
*   **Обработка ошибок:** Используйте специфические типы исключений, избегайте голого `except Exception`. Для критических компонентов (Risk Engine, Execution) предусматривайте механизмы повторных попыток (`retry`) и fallback-логику.
*   **Для стратегий разных классов:**
    *   **SFT-стратегии (Slow-Frequency):** строгое соблюдение SOLID (см. раздел 10). Используйте DI, протоколы, тесты с моками.
    *   **MFT-стратегии (Mid-Frequency) на Python:** допустимо упрощение DI и структуры, но с сохранением читаемости и тестируемости ключевых частей. Для производительных вычислений используйте `numpy`/`numba`.

## 3. Работа с кодом на Rust

*   **Стиль кода:** Следуйте стандартам `cargo fmt` и рекомендациям `clippy`. Запускайте `cargo clippy -- -D warnings`.
*   **Структура:** Весь Rust-код размещайте в workspace `crates/`. Каждый компонент должен быть отдельным крейтом:
    *   `eventbus/` — высокопроизводительная шина событий
    *   `risk-ledger/` — двойная бухгалтерия рисков
    *   `audit-chain/` — криптографическая цепочка аудита
    *   `execution-core/` — низкоуровневое исполнение
    *   `ffi/` — привязки к Python
    *   `common/` — общие типы и утилиты
*   **Безопасность:**
    *   В `execution-core` и `risk-ledger` минимизируйте использование `unsafe`. Каждый блок `unsafe` должен быть подробно задокументирован с объяснением его безопасности.
    *   Для детализированных ошибок используйте крейты `snafu` или `thiserror`.
*   **Взаимодействие с Python (FFI):**
    *   Любые изменения в крейте `ffi` должны сопровождаться обновлением Python-заглушек (`.pyi` файлов) в соответствующих местах `src/`.
    *   Аккуратно работайте с передачей строк и сложных структур через границу Python/Rust, избегайте утечек памяти.
*   **Для стратегий разных классов:**
    *   **HFT-стратегии (High-Frequency):** максимальная оптимизация, допустимо отступление от SOLID с обязательным ADR и профилированием. Используйте мономорфизацию (generics), избегайте динамической диспетчеризации (`dyn Trait`) в hot path. Применяйте `#[inline]` только после доказательства необходимости.
    *   **MFT-стратегии на Rust:** допустимо использование `dyn Trait`, если это не влияет на производительность критических сценариев. Отдавайте предпочтение Python-реализации, если нет веских причин для Rust.

## 4. Управление конфигурацией и секретами

*   **Секреты:** Никогда не хардкодьте API-ключи, пароли или токены. Используйте переменные окружения из файла `.env` (см. `.env.example`). В production обязательно применяйте **HashiCorp Vault**.
*   **Конфигурация:** Все конфигурации должны быть типизированными (Pydantic-модели в Python). Изменения в схемах конфигов должны быть обратно совместимыми или сопровождаться миграциями.

## 5. Тестирование (критически важно!)

### 5.1. Общий принцип
Главная цель тестирования — **уверенность в работоспособности системы**, особенно в критических сценариях. Не гоняйтесь за абстрактными процентами покрытия; вместо этого фокусируйтесь на качестве тестов и покрытии важных пользовательских и системных сценариев.

### 5.2. Ориентиры по покрытию (не жёсткие требования)
Установлены следующие *ориентиры* для различных категорий кода. Они служат для оценки, а не для формальной отчётности:

| Категория кода | Ориентир покрытия | Пояснение |
|----------------|-------------------|-----------|
| **Критические модули** (Risk Engine, Execution Core, Audit Chain, Event Bus) | >80% | Высокая степень уверенности в корректной работе и отсутствии регрессий. |
| **Бизнес-логика стратегий** (в зависимости от класса) | >70% для MFT/SFT, для HFT упор на property-based тесты и фаззинг | Стратегии должны быть протестированы на исторических данных и граничных случаях. |
| **Инфраструктурный код** (утилиты, адаптеры) | >60% | Покрываются основные пути использования. |
| **UI/Dashboard** | >50% | Покрываются ключевые пользовательские сценарии. |

### 5.3. Виды тестов по классам стратегий

| Класс стратегии | Приоритетные виды тестов | Комментарий |
|-----------------|--------------------------|-------------|
| **HFT** | Property-based тесты, фаззинг, интеграционные тесты с реплеем исторических данных | Основной упор на недетерминированные и нагрузочные тесты. Модульные тесты — второстепенны. |
| **MFT** | Unit-тесты, property-based для сложных расчётов, регрессионные тесты | Баланс между быстрыми модульными тестами и проверкой сложной логики. |
| **SFT** | Unit-тесты с моками, сквозные (end-to-end) тесты на исторических данных | Максимальная тестируемость благодаря DI и протоколам. |

### 5.4. Где писать тесты

*   **Модульные тесты (Python):** `tests/unit/`
*   **Модульные тесты (Rust):** внутри каждого крейта (стандартная папка `tests/` или модуль `tests`)
*   **Интеграционные тесты:** `tests/integration/` — проверка взаимодействия Python и Rust компонентов
*   **Сквозные тесты (E2E):** `tests/e2e/` — полные торговые сценарии на тестовых данных

### 5.5. Процесс

*   Перед коммитом убедитесь, что все тесты проходят: `pytest` для Python и `cargo test` для Rust.
*   При добавлении новой функциональности обязательно добавляйте тесты, покрывающие **основные сценарии её использования**.
*   Если вы обнаружили баг — сначала напишите тест, который его воспроизводит, а потом исправляйте код.

## 6. Архитектура и производительность

*   **Разделение ответственности:** Чётко соблюдайте разделение на Control Plane (Python, оркестрация) и Data Plane (Rust, производительность). Не дублируйте логику высоконагруженных компонентов (Event Bus, Ledger) в Python.
*   **Event Bus:** При работе с сообщениями всегда учитывайте приоритеты (система приоритетов из Phase 3). Для критических торговых сигналов используйте наивысший приоритет.
*   **Аудит:** Любое изменение состояния, влияющее на риск или баланс, должно проходить через `audit-chain` (криптографическая запись).
*   **Бенчмарки:**
    *   Бенчмарки производительности запускайте **вручную** перед релизом или при существенных изменениях в HFT-компонентах.
    *   Автоматический прогон в CI не производится из-за нестабильности результатов.
    *   Для регрессионного контроля используйте юнит-тесты с таймаутами (с запасом).

## 10. Адаптация SOLID под классы стратегий (HFT / MFT / SFT)

Проект поддерживает стратегии разных классов (HFT, MFT, SFT). Требования к SOLID и производительности зависят от класса. **Классификация из архитектурного плана v2.0 обязательна для понимания контекста.**

### 10.1. Классы стратегий и общий интерфейс

| Класс | Таймфрейм | Допустимая задержка | Приоритет | Язык реализации |
|-------|-----------|---------------------|-----------|-----------------|
| **HFT** | < 1 сек | Микросекунды – 1 мс | Скорость исполнения | Rust (отдельные крейты) |
| **MFT** | 1 сек – 5 мин | 1 мс – 100 мс | Баланс скорости и гибкости | Python (numpy/numba) или Rust при необходимости |
| **SFT** | > 5 мин | > 100 мс (секунды) | Расширяемость, тестируемость | Python (полный SOLID) |

**Единый интерфейс стратегии (независимо от класса):**
Каждая стратегия должна реализовывать общий базовый интерфейс, чтобы Control Plane мог управлять ими единообразно.

*   **Python (через Protocol):**
    ```python
    class Strategy(Protocol):
        def on_tick(self, tick: Tick) -> Optional[Signal]: ...
        def on_bar(self, bar: Bar) -> Optional[Signal]: ...
        def warmup_period(self) -> int: ...
        def required_capabilities(self) -> ExchangeCapabilities: ...
		
*	**Rust (через трейт):**
    ```rust
	pub trait Strategy: Send + Sync {
		fn on_tick(&mut self, tick: &Tick) -> Option<Signal>;
		fn on_bar(&mut self, bar: &Bar) -> Option<Signal>;
		fn warmup_period(&self) -> usize;
		fn required_capabilities(&self) -> ExchangeCapabilities;
	}
	
### 10.2. Применение SOLID по классам

#### **Для HFT-стратегий (Rust)**
*   **Главный приоритет — производительность.** Допускается *осознанное отступление* от SOLID, если это даёт значительный выигрыш в скорости (например, отказ от интерфейсов-трейтов в пользу мономорфизации, использование `unsafe` после профилирования).
*   **Обязательные условия при отступлении:**
    1.  Наличие **профилирования**, доказывающего необходимость оптимизации.
    2.  Подробный **комментарий в коде** со ссылкой на **ADR** (Architecture Decision Record), объясняющий компромисс.
    3.  Использование `#[inline]` только на действительно горячих функциях.
*   **Рекомендации:**
    *   Generics + мономорфизация вместо `dyn Trait`, если набор стратегий известен на этапе компиляции.
    *   Избегать лишних аллокаций в hot path (использовать `smallvec`, `arrayvec`).

#### **Для MFT-стратегий (Python, реже Rust)**
*   **Баланс между скоростью разработки и производительностью.** SOLID рекомендуется, но не требуется фанатично. Можно использовать упрощённый DI, не плодить абстракции без необходимости.
*   **Python:** для вычислений использовать `numpy`/`numba`, остальной код может быть более процедурным, но с читаемостью.
*   **Rust:** допустимо использование `dyn Trait`, если вызовы не слишком частые. Отдавайте предпочтение Python, если нет веских причин для Rust.

#### **Для SFT-стратегий (Python)**
*   **Строгое следование SOLID.** Это обеспечит лёгкость добавления новых идей, тестирования и долгосрочную поддерживаемость.
*   **Обязательно:**
    *   Использование `Protocol` для заменяемых зависимостей.
    *   Внедрение зависимостей через конструктор (DI).
    *   Тестирование с моками.
    *   Разделение на маленькие классы с единственной ответственностью.
*   **Пример:**
    ```python
    class SmaCrossStrategy:
        def __init__(self, fast: int, slow: int, risk_engine: RiskEngine):
            self.fast = fast
            self.slow = slow
            self.risk = risk_engine   # DI
			
### 10.3. Дополнительные архитектурные элементы

*   **ExchangeCapabilities:**  
    Каждая стратегия (любого класса) должна декларировать требуемые возможности биржи (типы ордеров, кредитное плечо и т.д.). При запуске проверяется совместимость с адаптером.

    ```python
    @dataclass
    class ExchangeCapabilities:
        order_types: frozenset[OrderType]
        max_leverage: Decimal
        min_order_size: Decimal
        supports_stops: bool
        # ...
		
*   **Batch API (FFI):**
	Для передачи тиков/баров из Python в Rust используйте пакетную обработку, чтобы уменьшить число вызовов и работу с GIL. Размер батча может быть адаптивным (например, настраиваться для каждой стратегии).

	```rust
	// Пример сигнатуры в FFI
	#[pyfunction]
	fn process_ticks(ticks: &[PyTick]) -> Vec<PySignal> {
		ticks.iter().map(process_one).collect()
	}

### 10.4. Документирование архитектурных решений (ADR)

Для каждого существенного компромисса (особенно в HFT-компонентах) создавайте **ADR** в папке `docs/adr/` со структурой:

*   **Заголовок**
*   **Дата** (в формате YYYY-MM-DD)
*   **Статус** (Принято / Заменено / Устарело)
*   **Контекст** (какая стратегия, класс, почему нужно отступить)
*   **Рассмотренные альтернативы**
*   **Решение** (что выбрали и почему)
*   **Последствия** (положительные и отрицательные)
*   **Связанные ADR** (если есть)

**Инструкция для Kilo Code:**
*   При обнаружении архитектурного решения, которое требует компромисса (например, отступление от SOLID в HFT-компонентах), **предложи создать ADR**.
*   Если пользователь согласен, **сгенерируй файл ADR** в папке `docs/adr/` с именем по шаблону `NNNN-kratkoe-opisanie.md` (где NNNN — следующий порядковый номер, начиная с 0001). Заполни все поля на основе контекста обсуждения.
*   Для новых ADR используй шаблон, который должен находиться в `docs/adr/0000-template.md`. Если шаблона нет, предложи его создать.
