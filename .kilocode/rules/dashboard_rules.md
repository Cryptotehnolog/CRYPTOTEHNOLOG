# Правила разработки панели управления CRYPTOTEHNOLOG

## 1. Назначение документа

Этот документ задаёт специальные правила для разработки панели управления `CRYPTOTEHNOLOG`.

Область действия:
- frontend панели;
- backend API панели;
- real-time слой панели;
- operator workflows;
- UX критических действий;
- модульная и фазируемая интеграция панели с backend платформы.

Этот документ **дополняет** общие правила проекта.  
Если правило панели уже и безопаснее описывает поведение dashboard-модуля, для панели применяется именно оно.

Базовый принцип панели:

> Панель управления — это интерфейс управления системой, а не декоративная оболочка над данными.

---

## 2. Цели панели

Панель должна:
- давать оператору быстрое понимание состояния платформы;
- безопасно отображать критические риски, деградации и инциденты;
- обеспечивать управляемый доступ к operator workflows;
- быть модульной;
- быть фазируемой;
- развиваться вместе с backend, не ломая общую архитектуру.

Панель **не должна**:
- имитировать готовность backend-функций;
- скрывать деградацию системы;
- смешивать критические действия с обычной навигацией;
- подменять реальную observability красивыми, но недостоверными виджетами.

---

## 3. Архитектурные принципы панели

### 3.1. Общая модель
Панель проектируется как `Platform Control Center`.

Она состоит из:
- глобального уровня статуса и команд;
- модульной рабочей области;
- контекстного слоя деталей, алертов и operator workflows.

### 3.2. Модульность
Все разделы панели проектируются как независимые модули.

Каждый модуль должен поддерживать один из статусов:
- `inactive`
- `read-only`
- `active`
- `restricted`

Модули должны подключаться независимо и не требовать переделки всей панели при включении нового backend-компонента.

### 3.3. Разделение слоёв
Нельзя смешивать:
- frontend UI;
- dashboard-specific state;
- facade/view model слой;
- внутреннюю доменную логику платформы.

Между UI и внутренними системными компонентами должен быть отдельный `dashboard API / facade layer`.

---

## 4. Правила для frontend

### 4.1. Технологический стек
Базовый стек панели:
- `React`
- `TypeScript`

Изменение этого стека должно быть отдельно обосновано.

### 4.2. Структура frontend-кода
- Frontend панели должен жить в отдельной верхнеуровневой папке проекта.
- Компоненты должны быть небольшими и предсказуемыми.
- Страницы не должны содержать тяжёлую доменную логику.
- Трансформации данных, derived state и orchestration выносятся из page-level компонентов в hooks, services или state-модули.

### 4.3. Обязательные UI-состояния
Каждый модуль и ключевой виджет должен корректно отображать:
- `loading`
- `empty`
- `stale`
- `error`
- `inactive`
- `read-only`
- `active`

Нельзя использовать один и тот же визуальный вид для:
- функции, которая не подключена;
- временной ошибки;
- отсутствия данных;
- потери соединения;
- ограничения по роли.

### 4.4. Принцип интерфейса
Панель должна быть операторской и системной по характеру.

Приоритет интерфейса:
1. состояние системы;
2. alerts;
3. pending approvals;
4. health;
5. события;
6. конфигурация и риск;
7. аналитика и вторичные графики.

---

## 5. Правила для dashboard API

### 5.1. API панели
Dashboard API должен быть отдельным слоем представления.

Он должен:
- агрегировать данные для UI;
- скрывать внутреннюю сложность backend;
- возвращать стабильные DTO;
- не протаскивать внутренние классы платформы напрямую в UI.

### 5.2. Типы API
API панели делится на:
- `REST snapshot endpoints`
- `WebSocket real-time stream`
- `controlled action endpoints`

### 5.3. DTO и модели представления
- Для панели использовать отдельные DTO/view models.
- DTO должны быть стабильными и семантически понятными.
- Нельзя использовать UI как место, где впервые определяется доменная семантика поля.
- Если внутренние backend-модели неудобны для UI, преобразование должно происходить в facade-слое.

### 5.4. Ошибки и состояния API
API панели обязано уметь различать:
- модуль ещё не подключён;
- функция недоступна в текущей фазе;
- данных нет;
- данные устарели;
- backend временно недоступен;
- действие запрещено;
- компонент деградирован.

---

## 6. Real-time и события

### 6.1. Общий принцип
Real-time слой панели должен быть построен на реальных событиях платформы, а не на имитации “живости”.

### 6.2. Что должно идти через real-time
Через WebSocket или эквивалентный поток должны передаваться:
- изменения состояния системы;
- alerts;
- health changes;
- critical events;
- operator requests;
- обновления статусов критических workflow.

### 6.3. Что не должно идти только через real-time
История, большие выборки и аналитические данные не должны зависеть только от WebSocket.  
Для них использовать snapshot/REST слой.

### 6.4. Устойчивость
Frontend панели обязан корректно переживать:
- разрыв соединения;
- повторное подключение;
- дубли событий;
- задержанные события;
- временную рассинхронизацию snapshot и stream.

Если данные больше не актуальны, UI должен явно это показать.

---

## 7. Operator workflows и критические действия

### 7.1. Базовый принцип
Опасные действия в панели не являются обычными кнопками.

Критические действия:
- `kill switch`
- `halt`
- `recovery`
- `risk override`
- критические config actions
- любые другие действия, влияющие на риск, состояние системы или исполнение

должны иметь отдельный управляемый UX-поток.

### 7.2. Что обязательно для критического действия
Каждое критическое действие должно иметь:
- причину;
- контекст;
- описание последствий;
- подтверждение;
- audit visibility;
- проверку допустимости действия;
- отражение operator identity.

### 7.3. Dual control
Если backend поддерживает `dual control`, UI обязан:
- показывать это как основной workflow;
- не скрывать pending approvals;
- не выдавать действие за немедленно выполненное, если оно только отправлено на подтверждение.

### 7.4. Запреты
- Нельзя размещать destructive actions среди обычных навигационных элементов.
- Нельзя позволять UI вызывать критическое действие напрямую, если backend требует operator gate или audit.
- Нельзя проектировать важные действия как “кнопку без последствий”.

---

## 8. Правила фазируемости

### 8.1. Общий принцип
Панель проектируется сразу как большая финальная система, но реализуется поэтапно.

### 8.2. Статусы модулей
Каждый модуль обязан иметь статус готовности:
- `inactive`
- `read-only`
- `active`
- `restricted`

### 8.3. Этапы подключения модуля
Каждый модуль проходит этапы:
1. shell
2. snapshot
3. live updates
4. controlled actions
5. stabilization

### 8.4. Запреты
- Нельзя делать вид, что модуль активен, если backend ещё не готов.
- Нельзя временно встраивать бизнес-логику в UI только потому, что backend-функция ещё не реализована.
- Нельзя проектировать early-stage модуль так, что позже его придётся переписывать целиком из-за отсутствия фазируемости.

---

## 9. UX-правила панели

### 9.1. Главный UX-критерий
Оператор должен понимать состояние платформы за 5-10 секунд.

### 9.2. Глобально видимые сущности
Из любого ключевого раздела должны быть видны:
- текущее состояние системы;
- критичность статуса;
- active alerts;
- pending approvals.

### 9.3. Иерархия важности
Системное состояние важнее графиков.  
Алерты важнее декоративных метрик.  
Pending approvals важнее вторичной аналитики.

### 9.4. Пустые и ограниченные состояния
- Пустое состояние должно объяснять, что именно отсутствует.
- Состояние `inactive` должно объяснять, что модуль предусмотрен, но ещё не подключён.
- Состояние `restricted` должно объяснять ограничение доступа.
- Состояние `stale` должно явно показывать потерю актуальности данных.

---

## 10. Наблюдаемость панели

### 10.1. Панель сама должна быть наблюдаемой
Панель и её API должны иметь:
- логирование ошибок;
- понятные состояния соединения;
- трассировку критических action flows;
- диагностику real-time проблем.

### 10.2. Ошибки UI
- Ошибки UI не должны молча скрываться.
- Критические ошибки отображения должны иметь операторски понятную форму.
- Нельзя превращать любой сбой панели в “просто пустой экран”.

---

## 10.3. Live-run, recovery и browser verification

### Базовый принцип
Live-run и recovery шаги панели должны выполняться так же дисциплинированно,
как и кодовые изменения.

Нельзя компенсировать отсутствие ясной диагностики длинными shell-командами,
массовым relaunch или “умными” скриптами, которые трудно контролировать.

### Порядок действий по умолчанию
Если требуется диагностика, recovery или browser verification, использовать
только такой порядок:
1. `status-check`
2. один короткий action step
3. чтение логов / HTTP-check
4. только потом следующий шаг

После каждого operational шага нужно:
- остановиться;
- кратко зафиксировать результат;
- не переходить автоматически к следующему действию без явного подтверждения состояния.

### Что запрещено
Для live-run / recovery шагов запрещено:
- склеивать `stop/start/check/logs` в одну длинную shell-команду;
- использовать длинные PowerShell-скрипты там, где можно сделать короткий шаг;
- использовать `Start-Process` по умолчанию для recovery;
- одновременно перезапускать backend и frontend без отдельной причины;
- смешивать code-fix и operational recovery в одном шаге;
- делать повторный relaunch “на всякий случай”, не проверив:
  - порт;
  - PID;
  - логи;
  - прямой HTTP-ответ.

### Что обязательно при `500` и live-сбоях
При ошибке `500`, `ErrorState`, failed live-check или browser сбое сначала нужно
локализовать источник:
- backend или frontend;
- прямой endpoint или proxy route;
- live-process residual или кодовый баг;
- наличие traceback в логах.

Только после этого допустим corrective action.

### Правило запуска процессов
Если процесс уже поднят и отвечает, его нельзя перезапускать без отдельной причины.

Если процесс не отвечает:
- сначала коротко проверить, кто слушает порт;
- затем отдельно остановить только проблемный процесс;
- затем отдельно поднять только нужный процесс;
- затем отдельно проверить логи и endpoint.

### Browser verification
Browser verification не должна начинаться до тех пор, пока:
- backend endpoint не отвечает корректно;
- есть понятное состояние логов;
- нет неразобранного live residual.

Если browser verification выявила ошибку:
- сначала локализовать failing request;
- затем проверить прямой backend endpoint;
- затем читать логи;
- только потом исправлять код или делать recovery.

---

## 11. Тестирование панели

### 11.1. Общий принцип
Для панели приоритетны не проценты покрытия, а ключевые операторские сценарии.

### 11.2. Что тестировать обязательно
- `inactive/read-only/active/restricted`
- loading/empty/error/stale
- reconnect real-time канала
- корректное отображение деградации
- alerts
- operator approval flows
- блокировку недоступных действий
- критические confirmations
- корректное отображение последствий действий

### 11.3. Dashboard API
Для dashboard API обязательно тестировать:
- стабильность DTO;
- маппинг из доменных моделей в view models;
- корректные коды и типы ошибок;
- отказоустойчивость при недоступности upstream-компонентов.

---

## 12. Правила развития панели

### 12.1. Подключение новых модулей
Новый модуль нельзя подключать напрямую в production-like UX без:
- места в общей архитектуре панели;
- состояния готовности;
- snapshot слоя;
- продуманного поведения ошибок;
- понимания operator impact.

### 12.2. Изменения существующих модулей
Изменение модуля не должно ломать:
- глобальную навигацию;
- общую модель статусов;
- глобальные alerts;
- operator workflows;
- фазовую модель подключения.

### 12.3. Документация
Для каждого крупного модуля панели должно быть понятно:
- что он показывает;
- откуда берёт данные;
- в каком он статусе;
- какие действия доступны;
- какие ограничения действуют.

---

## 13. Чего делать нельзя

- Подменять backend-готовность красивым UI.
- Размещать критические действия без подтверждений и operator context.
- Прятать ошибки канала или деградацию данных.
- Смешивать доменные модели платформы и view models панели.
- Строить весь UX вокруг вторичных графиков в ущерб системному состоянию.
- Делать панель как “торговый терминал”, если её задача — управление платформой.
- Делать новый модуль без учёта фазируемости.
- Делать вид, что отсутствие данных и ошибка системы — это одно и то же.

---

## 14. Практическое правило по умолчанию

Если есть выбор между:
- визуально эффектным решением;
- быстрым временным решением;
- надёжным, ясным и фазируемым решением,

для панели управления нужно выбирать надёжное, ясное и фазируемое решение.
