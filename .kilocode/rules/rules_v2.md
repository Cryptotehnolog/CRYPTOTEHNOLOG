# Правила проекта CRYPTOTEHNOLOG v2

## 1. Назначение документа

Этот документ определяет инженерные правила разработки проекта `CRYPTOTEHNOLOG`.

Цели правил:
- поддерживать надежность, отказоустойчивость и производительность системы;
- снижать архитектурный и операционный риск;
- сохранять единый стиль разработки;
- упрощать поэтапное развитие платформы;
- обеспечивать предсказуемое качество кода и изменений.

Базовый принцип проекта:

> Я не пишу код, который просто работает. Я строю системы, которые не падают.

Эти правила применяются ко всему проекту, если для отдельных модулей не определены более узкие специальные правила.

---

## 2. Общие принципы работы

### 2.1. Язык
- Все ответы, объяснения, комментарии к коду, docstrings, логи и сообщения об ошибках должны быть на **русском языке**, если явно не указано иное.
- Исключения допустимы только там, где этого требует внешний API, библиотека, стандарт протокола или формат данных.

### 2.2. Приоритеты проекта
При конфликте целей приоритеты такие:
1. Надежность
2. Отказоустойчивость
3. Корректность
4. Безопасность
5. Производительность
6. Расширяемость
7. Удобство разработки

### 2.3. Стиль инженерной работы
- Не использовать временные костыли как постоянные решения.
- Не скрывать архитектурные проблемы под UI, обертками или дублированием логики.
- Не вводить фиктивную функциональность, которая выглядит готовой, но не обеспечена backend.
- Любое значимое решение должно быть объяснимо с точки зрения надежности, производительности и сопровождения.
- Если найден баг, сначала нужно понять инвариант, который был нарушен, а не только устранить симптом.

### 2.4. Коммуникация по изменениям
- При архитектурно значимых изменениях сначала кратко объяснять решение, затем показывать код.
- При больших изменениях не перегружать ответ полными файлами без необходимости.
- При исправлении багов указывать причину, область влияния и риски регрессии.

## 3. Источники истины и управление зависимостями

### 3.1. Python-зависимости
- **Главный источник истины для Python-зависимостей — `pyproject.toml`.**
- `requirements.txt` и `requirements-prod.txt` считаются вторичными, вспомогательными файлами для совместимости, развёртывания или экспорта.
- Нельзя изменять `requirements*.txt` как самостоятельный основной источник зависимостей, игнорируя `pyproject.toml`.

### 3.2. Rust-зависимости
- Источник истины для Rust-зависимостей — соответствующие `Cargo.toml` внутри workspace.

### 3.3. Установка зависимостей
- Все Python-зависимости должны устанавливаться только в `.venv`.
- Нельзя устанавливать проектные Python-зависимости глобально.
- Для Rust использовать стандартные механизмы Cargo.

## 4. Структура проекта

### 4.1. Общие правила структуры
- Следовать существующей архитектуре проекта.
- Не создавать новые верхнеуровневые папки без явного архитектурного обоснования.
- Если новый крупный модуль действительно требует отдельной верхнеуровневой папки, это допускается после согласования.

### 4.2. Базовая логика Python
Следовать существующей структуре `src/cryptotechnolog/`:
- `config/` — конфигурация и настройки;
- `core/` — базовые системные компоненты;
- `risk/` — логика управления рисками;
- `analysis/` — анализ данных и рынка;
- `execution/` — исполнение ордеров;
- `models/` — модели данных;
- `intelligence/` — интеллектуальный слой;
- `strategy/` — торговые стратегии;
- `observability/` — мониторинг, health, метрики, dashboard-facing observability.

### 4.3. Rust
Весь Rust-код размещается в workspace `crates/`. Новые производительные компоненты должны жить как отдельные крейты, если это оправдано архитектурой.

## 5. Правила разработки на Python

### 5.1. Стиль и инструменты
- Следовать настройкам из `pyproject.toml`.
- Использовать:
  - `ruff`
  - `black`
  - `mypy`
- Перед значимыми изменениями и перед коммитом проверять релевантные части проекта через линтер и типизацию.

### 5.2. Типизация
- Для нового и изменяемого кода использовать аннотации типов обязательно.
- Для сложных импортов и избежания циклических зависимостей использовать `TYPE_CHECKING`.
- Не вводить неявные структуры там, где уместны типизированные модели, dataclass или protocol.

### 5.3. Ошибки и исключения
- Использовать специфические типы исключений.
- Избегать голого `except Exception`, кроме граничных слоёв, где ошибка:
  - логируется;
  - преобразуется;
  - не теряется бесконтрольно.
- Для критических компонентов предусматривать retry, fallback или безопасную деградацию, если это архитектурно оправдано.

### 5.4. Структура кода
- Не перегружать модули несвязанной логикой.
- Не смешивать доменную модель, инфраструктурный код и код представления.
- Для расширяемых частей использовать протоколы, DI и небольшие абстракции там, где это реально упрощает сопровождение.

## 6. Правила разработки на Rust

### 6.1. Стиль и инструменты
- Использовать `cargo fmt`.
- Следовать рекомендациям `clippy`.
- Для нового и изменяемого кода стремиться к прохождению `cargo clippy -- -D warnings`, если это применимо к модулю.

### 6.2. Безопасность
- Минимизировать использование `unsafe`.
- Каждый `unsafe` блок должен быть:
  - обоснован;
  - локализован;
  - задокументирован;
  - объяснён с точки зрения безопасности.

### 6.3. Ошибки
- Использовать типизированные ошибки.
- Предпочитать `thiserror` или `snafu`, если это соответствует стилю модуля.

### 6.4. FFI и интеграция с Python
- Любые изменения в FFI-слое должны сопровождаться обновлением Python-оберток и type stubs, если они существуют.
- Передача данных через границу Python/Rust должна быть явной, безопасной и без скрытых утечек памяти.
- Для интенсивных сценариев предпочитать пакетную обработку вместо частых мелких вызовов.

## 7. Архитектурные принципы

### 7.1. Разделение плоскостей
- `Control Plane` и orchestration-логика должны оставаться в Python, если нет явной причины переносить их в Rust.
- `Data Plane` и критичные по производительности части должны жить в Rust или в строго оптимизированных Python-компонентах, если это достаточно.

### 7.2. Не дублировать критичную логику
- Нельзя дублировать в Python критичную логику, уже реализованную в Rust, только ради удобства.
- Нельзя дублировать risk-, audit-, ledger- или event-логику в UI или API-слоях.

### 7.3. Event-driven подход
- Изменения состояния системы и критические операционные события должны быть выражены через события и аудит.
- При работе с событиями учитывать приоритеты, очереди, задержки, backpressure и отказоустойчивость.

### 7.4. Аудит
- Любое действие, влияющее на риск, состояние системы, исполнение, баланс, конфигурацию или operator workflow, должно быть наблюдаемым и аудируемым.
- Если действие не может быть надёжно отражено в audit trail, его реализация должна считаться незавершённой.

## 8. Конфигурация и секреты

### 8.1. Секреты
- Никогда не хардкодить ключи, токены, пароли и другие секреты.
- Для локальной разработки допустимы `.env` и `.env.example`.
- Для production и чувствительных окружений использовать `Infisical` или другой утверждённый менеджер секретов.

### 8.2. Конфигурация
- Конфигурации должны быть типизированными.
- Изменения в схемах конфигурации должны быть обратно совместимыми или сопровождаться миграциями.
- Конфигурация должна валидироваться до применения.
- Если конфиг влияет на критическое поведение системы, изменение должно быть наблюдаемым и аудируемым.

## 9. Тестирование

### 9.1. Общий принцип
Главная цель тестирования — уверенность в работоспособности системы в критических сценариях, а не формальный процент покрытия.

### 9.2. Приоритеты тестов
Приоритет тестирования:
1. критические системные инварианты;
2. рисковые и операционные сценарии;
3. интеграция между компонентами;
4. регрессии найденных багов;
5. затем общее покрытие.

### 9.3. Практические требования
- При добавлении новой функциональности добавлять тесты на основные сценарии использования.
- При исправлении бага сначала добавить воспроизводимый тест или хотя бы формализованный сценарий воспроизведения, затем исправлять код.
- Модульные, интеграционные и E2E тесты должны использоваться по назначению, без подмены одного вида тестов другим.

### 9.4. Ориентиры покрытия
Ориентиры покрытия:
- критические модули: высокий приоритет тестов;
- инфраструктурный код: покрывать основные сценарии;
- UI/dashboard: покрывать ключевые операторские сценарии и критические действия.

Проценты покрытия можно использовать как ориентир, но не как самоцель.

## 10. HFT / MFT / SFT

### 10.1. Общий принцип
Требования к архитектуре, абстракциям и производительности зависят от класса стратегии.

### 10.2. HFT
- Главный приоритет — производительность.
- Допускается осознанное отступление от SOLID, если это доказано профилированием.
- Каждый такой компромисс должен быть объяснён и, при необходимости, оформлен через ADR.

### 10.3. MFT
- Приоритет — баланс скорости разработки и производительности.
- Допустимы упрощённые абстракции, если это не ухудшает читаемость и тестируемость ключевых частей.
- Для численных вычислений использовать подходящие инструменты (`numpy`, `numba`, Rust при необходимости).

### 10.4. SFT
- Для SFT-логики требуется наиболее строгая архитектурная дисциплина:
  - DI;
  - небольшие классы;
  - protocol-интерфейсы;
  - тестируемость;
  - ясное разделение ответственности.

---

## 11. Правила для панели управления и dashboard-модулей

### 11.1. Общий принцип
Панель управления — это часть платформы, а не отдельная декоративная витрина.

Её приоритеты:
- прозрачность состояния системы;
- безопасные operator workflows;
- наблюдаемость;
- фазируемость;
- отсутствие фальшивой готовности.

### 11.2. Архитектура панели
- Frontend панели должен жить в отдельном модуле проекта.
- Backend API панели должен быть отдельным слоем представления, а не прямым доступом к внутренним классам.
- Панель должна быть модульной и фазируемой:
  - `inactive`
  - `read-only`
  - `active`
  - `restricted`

### 11.3. UI и operator workflows
- Нельзя показывать интерактивные элементы для функций, которых backend ещё не поддерживает безопасно.
- Критические действия не должны быть обычными кнопками без контекста, причин и подтверждения.
- Глобальные индикаторы состояния системы, alerts и pending approvals должны быть видны из любого ключевого раздела.

### 11.4. Real-time
- Real-time обновления должны строиться на событиях и устойчиво переживать reconnect, stale state и деградацию канала.
- При потере актуальности данных интерфейс обязан это явно показывать.

## 12. Производительность

### 12.1. Общие правила
- Оптимизировать только после понимания узкого места.
- Не делать сложные оптимизации “на всякий случай”.
- Любая оптимизация, ухудшающая читаемость или архитектуру, должна быть оправдана.

### 12.2. Бенчмарки
- Производительные бенчмарки запускать вручную перед релизом или при существенных изменениях в hot path.
- Для регрессионного контроля использовать тесты, метрики и замеры там, где это практично.

## 13. Окружение и запуск

### 13.1. Python
- Работать только через `.venv`.
- Не использовать глобальный `pip install` для проектных библиотек.
- Изменения зависимостей сначала отражать в `pyproject.toml`.

### 13.2. Docker
- Docker использовать для инфраструктуры и сервисов окружения.
- Код приложения в процессе разработки запускать локально, если это ускоряет цикл обратной связи.

### 13.3. Operational discipline для live-run и recovery
- Live-run, recovery и process management выполнять только короткими шагами.
- Нельзя объединять в одну shell-команду:
  - остановку процесса;
  - запуск процесса;
  - чтение логов;
  - HTTP-проверку;
  - browser verification.
- После каждого operational шага нужно сначала зафиксировать результат, и только потом переходить к следующему.
- Нельзя использовать `Start-Process` и длинные PowerShell-цепочки как поведение по умолчанию для recovery.
- При `500`, failed endpoint или зависшем процессе сначала выполняется локализация причины:
  - кто слушает порт;
  - какой PID жив;
  - что в логах;
  - прямой endpoint падает или только proxy/UI.
- Только после локализации причины допустим stop/start или code-fix.
- Нельзя одновременно делать code changes и live recovery в одном и том же operational шаге без явной необходимости.

## 14. Git, CI и качество изменений

### 14.1. Коммиты
- Коммиты должны быть осмысленными.
- Перед коммитом должны проходить релевантные проверки качества.

### 14.2. CI
- Изменения не считаются готовыми, если они ломают CI, критические тесты или базовые проверки качества.
- Если определённый тип теста не прогоняется автоматически, это должно быть явно осознано, а не случайно упущено.

## 15. Документация

### 15.1. Код
- Сложные Python-функции и модули документировать через docstrings.
- Публичные Rust API документировать через `///`.

### 15.2. Проект
- При изменении ключевой архитектуры, способов запуска или модулей обновлять соответствующую документацию.
- Для операционных процедур использовать runbooks.
- Для значимых архитектурных компромиссов использовать ADR.

---

## 16. ADR

Для важных архитектурных решений, компромиссов производительности или отклонений от стандартного подхода создавать ADR в `docs/adr/`.

Структура ADR:
- заголовок;
- дата;
- статус;
- контекст;
- альтернативы;
- решение;
- последствия;
- связанные ADR.

---

## 17. Чего делать нельзя

- Хардкодить секреты.
- Дублировать критичную бизнес-логику между слоями.
- Подменять отсутствующую backend-функцию красивым UI.
- Оставлять неочевидный `unsafe` без документации.
- Вводить новые зависимости без отражения в главном источнике истины.
- Маскировать ошибки через избыточные `try/except`.
- Притворяться, что система production-ready, если не закрыты audit, risk или operator invariants.
- Ломать модульность ради краткосрочного удобства.

---

## 18. Практическое правило по умолчанию

Если есть сомнение между:
- быстрым решением;
- красивым решением;
- надёжным и расширяемым решением,

нужно выбирать надёжное и расширяемое решение, если это не создаёт несоразмерной сложности без доказанной пользы.
