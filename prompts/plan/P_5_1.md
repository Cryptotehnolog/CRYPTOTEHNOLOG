## 🎯 СМЫСЛ ЭТАПА

Фаза `P_5_1` — это не новая feature-phase.

Это этап жёсткой инженерной консолидации платформы после `v1.5.0`, цель которого:
- устранить расхождения между `prompt -> plan -> code -> runtime`;
- перевести платформу из состояния “сильные модули по отдельности” в состояние “единый, однозначный, production-grade runtime”;
- убрать архитектурную двусмысленность перед дальнейшим развитием торговых слоёв.

**Главный принцип этапа:**

> Следующий шаг проекта — не расширять платформу новыми trading-features, а сначала сделать текущую платформу инженерно состоятельной для работы с реальными деньгами.

---

## ✅ ЧТО УЖЕ ГОТОВО К НАЧАЛУ ЭТАПА

После Фаз `P_1–P_5` в проекте уже есть:

- инфраструктурный foundation;
- Control Plane;
- Enhanced Event Bus;
- Config Manager;
- новый `Risk Engine`;
- `RiskLedger` как source of truth;
- `TrailingPolicy` с инвариантами;
- controlled coexistence нового и legacy risk-path;
- phase summary `P_5_RESULT.md`;
- ADR:
  - `0021-risk-ledger-source-of-truth.md`
  - `0022-trailing-policy-risk-engine-invariant.md`
  - `0023-risk-engine-controlled-coexistence.md`
- два аудита, которые уже зафиксировали основной архитектурный риск:
  - сильное доменное ядро;
  - но опасные расхождения между prompt/plan/code/runtime.

---

## ❗ ПОЧЕМУ НУЖНА ИМЕННО ЭТА ФАЗА

По итогам Фазы 5 проект находится в промежуточном состоянии:

### Что уже сильно
- доменная модель риска;
- инварианты `RiskLedger` и `TrailingPolicy`;
- event-driven foundation;
- state-driven системное мышление;
- test foundation.

### Что остаётся опасным
- главный runtime entrypoint всё ещё не production-grade;
- новый `Risk Engine` ещё не стал единственным активным risk path;
- в bootstrap остаётся controlled coexistence;
- есть import-time side effects и singleton-driven скрытая инициализация;
- есть version/config/runtime drift;
- observability и startup policy не доведены до уровня production truth;
- часть event/runtime contracts реализована уже, но ещё не полностью выровнена с phase-level intent.

---

## 🧱 ЦЕЛЬ ФАЗЫ

Собрать **один официальный production runtime платформы**, который:

1. поднимается через единый composition root;
2. имеет один активный risk path;
3. не создаёт скрытого runtime state на import-time;
4. не использует тихие fallback-path для критичных компонентов;
5. имеет однозначные startup / readiness / health / shutdown semantics;
6. публикует и наблюдает критичный runtime/risk vocabulary;
7. отражает реальное состояние платформы в коде, документации и ADR.

---

## 🚫 ЧТО НЕ ЯВЛЯЕТСЯ ЦЕЛЬЮ ЭТАПА

Фаза `P_5_1` **не должна** расползтись в новую feature-line.

Вне scope этой фазы:
- новые trading strategies;
- расширение dashboard сверх минимально нужной observability;
- глубокая funding integration как полноценный risk gate, если она не блокирует production alignment;
- возврат к PyO3/FFI-only архитектуре Event Bus, если это не критично для runtime-consistency;
- новые performance-оптимизации “на будущее” без подтверждённого bottleneck;
- усложнение correlation/funding/statistical logic.

---

## 🔒 ОБЯЗАТЕЛЬНЫЕ ИНЖЕНЕРНЫЕ ПРИНЦИПЫ ФАЗЫ

### 1. Один production runtime
У платформы должен быть один официальный production entrypoint.

### 2. Один активный risk path
В одном runtime не может быть двусмысленного coexistence двух risk-контуров.

### 3. Fail-fast важнее silent fallback
Критичные зависимости нельзя “тихо ослаблять”.

### 4. Import должен быть безопасным
Импорт пакета не должен создавать runtime state, подключения и скрытые side effects.

### 5. Runtime contracts важнее пожеланий
Если phase-level замысел важен для runtime, он должен быть:
- либо реализован;
- либо официально пересмотрен через ADR и документацию.

### 6. Observability должна отражать правду
Health/readiness/version/active risk path/config identity должны показывать фактическое состояние системы.

| Задача | Смысл |
|--------|-------|

- settings/config;
- logging;
- database / redis;
- event bus;
- state machine / controller / watchdog;
- metrics / health;
- listeners;
- новый risk runtime;

- shutdown lifecycle.

## 🧪 КЛЮЧЕВЫЕ ТЕСТОВЫЕ СЦЕНАРИИ

### Bootstrap
- успешный startup через production entrypoint;
- startup с отсутствующей критичной зависимостью;
- startup в допустимом degraded path;
- корректный shutdown всех компонентов.

### Risk Path
- активен только новый `RiskEngine`;
- legacy path не регистрируется через production registry;
- невозможна двойная публикация risk-events из двух path.

### Import Safety
- импорт пакета не поднимает настройки/подключения/runtime;
- тестовый bootstrap можно собрать явно и изолированно;
- import order не влияет на runtime state.

### Metadata
- версия в package/settings/health/runtime совпадает;
- health/readiness показывают реальную release identity;
- active risk path и bootstrap mode видны в metadata.

### Event Contracts
- pre-trade reject публикуется корректно;
- drawdown escalation публикуется корректно;
- trailing moved / blocked публикуются корректно;
- state-sync события публикуются корректно;
- critical runtime failures не скрываются fallback-path.

## 🧾 DEFINITION OF DONE

Фаза `P_5_1` считается завершённой только если одновременно выполнено всё ниже:

1. У платформы есть один официальный production entrypoint и один composition root.
2. Production runtime поднимает систему через явный bootstrap lifecycle.
3. В production runtime активен ровно один risk path.
4. Legacy coexistence больше не остаётся рабочей runtime-двусмысленностью.
5. Импорт пакетов не создаёт скрытого bootstrap и критичных side effects.
6. Critical bootstrap failures приводят к fail-fast или явной деградации, а не к тихому fallback.
7. Version/config/runtime metadata согласованы и идут из одного источника истины.
8. Критичные risk/event contracts реализованы или официально пересмотрены.
9. Startup, readiness, health, shutdown и degradation наблюдаемы и аудируемы.
10. Документация и ADR отражают фактическое runtime-state платформы.
11. Integration tests проверяют реальный startup/runtime behavior, а не только отдельные модули.

## 🚀 РЕЗУЛЬТАТ ФАЗЫ

После завершения `P_5_1` платформа должна перейти в состояние:

- не “несколько сильных модулей и контролируемое coexistence”;
- а “один production-grade runtime с однозначной архитектурой”.
