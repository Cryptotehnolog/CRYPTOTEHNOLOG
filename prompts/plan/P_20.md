## 🎯 СМЫСЛ ФАЗЫ

`P_20` открывается как следующая нормализованная линия после:

- `Validation Foundation`
- `Paper Trading Foundation`
- active `Production Alignment` / runtime discipline

Фаза открывается узко как:

`Backtesting / Replay Foundation`

А не как:

- analytics / reporting platform;
- paper/live comparison platform;
- dashboard / operator surface;
- broader research / replay / optimization lab;
- hidden simulation owner для `Execution` / `OMS`.

Фактический смысл фазы:

- нормализовать уже существующий deferred `backtest` contour в отдельную authoritative foundation line;
- ввести explicit replay/backtest runtime boundary в текущей package discipline;
- зафиксировать typed replay/backtest contracts и event vocabulary;
- реализовать deterministic ingestion historical inputs без scope inflation в analytics, dashboard или research platform;
- удержать границы с уже закрытыми `Validation` и `Paper` foundations.

Главный принцип opening truth:

> `P_20` строит foundation для узкого replay/backtest contour, который
> replay-ит и record-ит existing truths,
> но не переприсваивает ownership уже реализованных runtime layers
> и не маскирует под себя broader analytics / simulation platform.

## 🧱 АРХИТЕКТУРНЫЕ ГРАНИЦЫ

`P_20` должна сохранить уже установленную ownership discipline.

### 1. Backtesting / Replay не подменяет Validation

`Validation`:

- владеет narrow review / evaluation truth;
- не становится submodule будущего replay layer.

`P_20` может использовать validation truth как один из replay inputs,
но не получает ownership над:

- review semantics;
- validation lifecycle truth;
- broader evaluation platform semantics.

### 2. Backtesting / Replay не подменяет Paper

`Paper`:

- владеет narrow rehearsal / controlled-simulation truth;
- не становится submodule будущего replay layer.

`P_20` может coexist-ить рядом с paper contour,
но не получает ownership над:

- rehearsal semantics;
- paper runtime state;
- broader controlled-simulation semantics уже закреплённые в `P_19`.

### 3. Backtesting / Replay не подменяет Execution и OMS

`Execution`:

- владеет operational execution intent / bridge semantics.

`OMS`:

- владеет order-state / lifecycle truth.

`P_20` может replay-ить или record-ить existing truths,
но не получает ownership над:

- execution decisions;
- order lifecycle;
- simulated order-management platform semantics.

### 4. Backtesting / Replay не подменяет Manager

`Manager`:

- владеет workflow-coordination truth.

`P_20` не должна становиться hidden workflow owner
только потому, что replay path может использовать manager-originated truths.

### 5. Future analytics / dashboard / research lines не должны маскироваться под P_20

Даже если future backtesting-supporting lines later могут использовать replay truth,
в рамках `P_20` она не должна автоматически превращаться в:

- analytics / reporting platform;
- dashboard-led operator surface;
- paper/live comparison platform;
- optimization / parameter-search platform;
- research / simulation lab.

---

## 🔌 RUNTIME / BOUNDARY DISCIPLINE

Фаза должна сохранить установленную runtime discipline проекта:

- explicit runtime entrypoint;
- явный `start()` / `stop()`, если runtime действительно вводится на этой фазе;
- отсутствие hidden bootstrap;
- query/state-first surface;
- operator-visible diagnostics только в той мере, в какой они нужны foundation contour;
- явная degraded/readiness truth, если runtime интегрируется в composition root.

Composition root не должен:

- вручную моделировать replay domain context;
- брать на себя ownership replay semantics;
- переприсваивать ownership `Validation`, `Paper`, `Execution`, `OMS`, `Manager`.

Bootstrap должен только:

- собрать replay runtime, если он действительно вводится;
- wiring-ить existing typed truths;
- встроить diagnostics в общую runtime/health truth без hidden behavior.
