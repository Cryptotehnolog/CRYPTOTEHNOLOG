## 🎯 СМЫСЛ ФАЗЫ

`P_11` открывает первую отдельную opportunity / selection-line поверх уже
завершённых `Signal Foundation`, `Strategy Foundation` и `Execution Foundation`.

Фактическая задача фазы:

- ввести opportunity / selection layer как самостоятельный production-compatible consumer contour;
- зафиксировать typed opportunity / selection contracts и explicit runtime boundary;
- реализовать минимальную ranking / selection candidate semantics поверх уже существующих typed truths;
- встроить runtime в текущую runtime discipline без hidden bootstrap;
- не превращать фазу в OMS, portfolio governor, strategy manager, meta-orchestration или supervisor line.

Главный принцип фазы:

> `P_11` строит foundation для узкого opportunity / selection contour, а не широкую order-management, portfolio или orchestration platform.

## 🧱 АРХИТЕКТУРНЫЕ ГРАНИЦЫ

### 1. Opportunity / selection layer является consumer-ом existing truths

Selection layer:

- потребляет уже существующие typed truths;
- не владеет `market_data`, `analysis`, `intelligence`, `signals`, `strategy`, `execution` или `risk` truth;
- не подменяет execution layer;
- не подменяет portfolio governance;
- не подменяет future meta/orchestration line.

### 2. Bootstrap не владеет selection semantics

Composition root:

- wiring-ит existing runtime contours;
- передаёт selection layer только existing typed truths;
- не собирает selection contracts вместо selection layer;
- не реализует ranking / selection logic внутри bootstrap.

### 3. Event truth не подменяет state/query truth

Selection layer может публиковать typed events, но основная truth должна оставаться:

- query/state-first;
- operator-visible;
- deterministic.

### 4. Runtime-time truth живёт в runtime

Expiry / invalidation / degraded semantics определяются в runtime, а не маскируются под dataclass-level temporal truth.

### 5. Opportunity / selection layer не подменяет later orchestration lines

Первый selection contour не должен автоматически тянуть:

- full `MetaClassifier`;
- multi-strategy conflict resolution;
- `StrategyManager`;
- portfolio-level supervision;
- advanced execution coordination.

Foundation сначала фиксирует ownership, contracts и runtime boundary.

## 🧩 RUNTIME / BOUNDARY DISCIPLINE

`P_11` строится по тем же дисциплинам, что уже закреплены в `P_7-P_10`:

- explicit runtime entrypoint;
- no hidden bootstrap;
- no import-time subscriptions;
- diagnostics sink как operator-visible truth;
- narrow ingest boundary;
- честное разделение structural invariants и runtime-time semantics.

Итоговый boundary фазы должен быть таким:

- selection runtime получает existing typed truths через явный ingest path;
- selection layer сама внутри собирает selection context;
- bootstrap only wires upstream truths into runtime;
- query/state surface позволяет честно наблюдать selection state без необходимости читать только events.
