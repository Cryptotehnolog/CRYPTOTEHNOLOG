## 🎯 ЦЕЛЬ ФАЗЫ

Реализовать production-ready `Risk Engine v4.4`, который:

1. Проверяет допустимость сделки до исполнения
2. Управляет размером позиции через `R-unit`
3. Контролирует correlation / exposure / drawdown
4. Управляет открытым риском через `TrailingPolicy`
5. Синхронизирует риск позиций через `RiskLedger`
6. Реагирует на рыночные и системные деградации
7. Публикует события риска и оставляет полный audit trail

---

## ⚠️ КРИТИЧЕСКИЕ ИНВАРИАНТЫ ФАЗЫ

### Глобальные инварианты
- Ни одна сделка не проходит без `RiskEngine.check_trade()`
- `RiskEngine` не одобряет сделки вне разрешённых состояний `State Machine`
- `TrailingPolicy` не может увеличить риск позиции
- Стоп-цена монотонна: не отдаляется от цены в сторону роста риска
- Нельзя двигать стоп без синхронного обновления `RiskLedger`
- При ошибке `RiskLedger` изменение стопа блокируется
- Все критические risk-события публикуются в `Event Bus`
- Все изменения риска должны быть аудируемы

### Инварианты по трейлингу
- `TrailingPolicy` принадлежит `RiskEngine`, а не стратегии
- `TrailingPolicy` работает по состояниям `INACTIVE -> ARMED -> ACTIVE -> EMERGENCY -> TERMINATED`
- `EMERGENCY` режим жёстче обычного и отключает soft/structural logic
- Structural trailing разрешён только при подтверждённых рыночных условиях
- При `HALT` трейлинг не выполняется
- При `SURVIVAL` или риск-деградации трейлинг переходит в `EMERGENCY`

### Инварианты по ledger
- `RiskLedger` — единый источник истины по риску позиций
- После открытия позиции риск должен быть зарегистрирован
- После движения стопа риск должен быть пересчитан
- После закрытия позиции риск должен быть освобождён
- Любое расхождение между позицией и ledger считается ошибкой системы

## 🧱 ПОРЯДОК РЕАЛИЗАЦИИ БЕЗ ОШИБОК

### Этап A. Сначала доменные модели и инварианты
Сначала реализовать:
- `models.py`
- `position_sizing.py`
- базовые тесты
- базовые enum/contract типы

**Почему:**  
без этого нельзя безопасно строить trailing и ledger.

### Этап B. Потом `RiskLedger`
Сначала нужен ledger как единый источник истины.

**Почему:**  
иначе trailing получится полусамостоятельным и нарушит инварианты.

### Этап C. Потом `TrailingPolicy`
Реализовывать только после того, как есть:
- модель позиции
- расчёт риска
- ledger update path

**Почему:**  
`No stop move without RiskLedger sync` — обязательный инвариант.

### Этап D. Потом `DrawdownMonitor` и `Correlation`
После того как есть базовый risk snapshot и portfolio state.

### Этап E. Потом `RiskEngine`
Собирать orchestration только после того, как базовые модули уже автономно протестированы.

### Этап F. Потом `FundingManager`
Последним из функциональных кусков, если он не блокирует основной путь фазы.

## 🧪 КЛЮЧЕВЫЕ ТЕСТОВЫЕ СЦЕНАРИИ

### Position sizing
- корректный размер позиции при обычном входе
- reject при `entry == stop`
- reject при превышении `max_r_per_trade`
- корректный расчёт `risk_usd` и `risk_r`

### Drawdown
- soft limit warning
- hard limit reject/halt path
- velocity killswitch при потере `2R` за 10 сделок

### Correlation
- reject при correlation > limit
- reject по correlation group limit
- approve при допустимой диверсификации

### RiskLedger
- register position
- update risk after stop move
- release on close
- aggregate total risk
- timeout/error blocks trailing move

### TrailingPolicy
- `INACTIVE -> ARMED -> ACTIVE`
- выбор `T1/T2/T3/T4` по `pnl_r`
- `EMERGENCY` режим при системной деградации
- стоп монотонен
- риск не растёт
- стоп не уходит “за рынок”
- при ledger failure движение блокируется

### RiskEngine
- reject вне разрешённых состояний
- approve в `TRADING/DEGRADED`
- reject по drawdown
- reject по velocity killswitch
- обработка `ORDER_FILLED`
- обработка `BAR_COMPLETED`
- публикация событий риска

## ⚠️ ОСОБЫЕ ПРЕДУПРЕЖДЕНИЯ ДЛЯ РЕАЛИЗАЦИИ

- Нельзя начинать с `engine.py`, пока не готовы доменные модели, ledger и trailing.
- Нельзя упрощать `TrailingPolicy` до “обычного ATR trailing”.
- Нельзя позволять движению стопа проходить без ledger sync.
- Нельзя смешивать domain state позиции и transport payload из Event Bus.
- Нельзя делать `FundingManager` настолько большим, чтобы он блокировал завершение основной risk-функциональности.
- Если часть `FundingManager` не успевает в фазу без потери качества, лучше сделать его минимально работоспособным, чем ломать основную risk-архитектуру.
- Если реализация `TrailingPolicy` начинает разрастаться без тестов, нужно остановиться и дописать тесты до продолжения.

## 📝 ASSUMPTIONS

- `State Machine`, `Event Bus`, `Config Manager`, `Database`, `Metrics` уже доступны как зависимости.
