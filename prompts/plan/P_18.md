## 🎯 СМЫСЛ ФАЗЫ

`P_18` доведена до closure-ready состояния как первая отдельная линия
`Validation Foundation`
поверх уже завершённых:

- `Execution Foundation`
- `OMS Foundation`
- `Opportunity / Selection Foundation`
- `Strategy Orchestration / Meta Layer`
- `Position Expansion Foundation`
- `Portfolio Governor / Capital Governance Foundation`
- `Protection / Supervisor Foundation`
- `Strategy Manager / Workflow Foundation`
- active `Risk Engine` / runtime discipline

Фактическая задача фазы:

- ввести validation layer как самостоятельный production-compatible review contour;
- зафиксировать typed validation contracts и explicit runtime boundary;
- реализовать narrow validation semantics поверх already existing typed truths;
- встроить validation runtime в текущую runtime discipline без hidden bootstrap;
- не превращать фазу в analytics platform, full performance reporting line, backtesting engine, paper trading system или dashboard-led operations surface.

Главный принцип фазы:

> `P_18` строит foundation для узкого validation/review contour, который
> оценивает existing truths, а не переприсваивает ownership уже реализованных runtime layers.

## 🧱 АРХИТЕКТУРНЫЕ ГРАНИЦЫ

`P_18` должна сохранить уже установленную ownership discipline.

### 1. Validation layer оценивает existing truths, а не подменяет их

Validation layer:

- потребляет existing typed truths;
- строит narrow review / validation state;
- не переписывает задним числом ownership соседних runtime layers.

### 2. Validation не подменяет Execution и OMS

`Execution`:

- владеет `order-intent` truth;
- не становится validation submodule.

`OMS`:

- владеет `order-state / lifecycle` truth;
- не становится validation-owned registry.

Validation может читать их truth, но не получает ownership над execution или order lifecycle.

### 3. Validation не подменяет Opportunity / Orchestration / Position Expansion

`Opportunity`:

- владеет selection truth.

`Orchestration`:

- владеет narrow meta-decision truth.

`Position Expansion`:

- владеет add-to-position contour.

Validation может оценивать уже полученные truths, но не объявляет себя owner-ом ranking, arbitration или expansion policy.

### 4. Validation не подменяет Portfolio Governor / Protection / Manager

`Portfolio Governor`:

- владеет capital-admission truth.

`Protection`:

- владеет narrow supervisory truth.

`Manager`:

- владеет narrow workflow-coordination truth.

Validation может строить review state поверх этих truths,
но не получает ownership над:

- capital governance;
- supervisory decisions;
- workflow coordination;
- broader control-plane semantics.

### 5. Future analytics / backtesting / paper-trading lines не должны маскироваться под Validation

Даже если future validation-supporting lines later могут использовать validation truth,
в рамках `P_18` она не должна автоматически превращаться в:

- analytics / performance platform;
- backtesting engine;
- paper trading system;
- reporting / benchmarking platform;
- dashboard-led operator surface.

---

## 🔌 RUNTIME / BOUNDARY DISCIPLINE

Фаза должна сохранить установленную runtime discipline проекта:

- explicit runtime entrypoint;
- явный `start()` / `stop()`;
- отсутствие hidden bootstrap;
- query/state-first surface;
- operator-visible diagnostics;
- явная degraded/readiness truth.

Composition root не должен:

- собирать validation domain context;
- вручную моделировать validation semantics;
- брать на себя ownership соседних layer truths.

Bootstrap должен только:

- собрать validation runtime;
- wiring-ить upstream typed truths;
- встроить diagnostics в общую runtime/health truth.
