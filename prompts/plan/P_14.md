## 🎯 СМЫСЛ ФАЗЫ

`P_14` открывает первую отдельную линию `Portfolio Governor / Capital Governance Foundation`
поверх уже завершённых:

- `Execution Foundation`
- `Opportunity / Selection Foundation`
- `Strategy Orchestration / Meta Layer`
- `Position Expansion Foundation`
- active `Risk Engine` / runtime discipline

Фактическая задача фазы:

- ввести `portfolio governor` как самостоятельный production-compatible consumer contour;
- зафиксировать typed portfolio-governor contracts и explicit runtime boundary;
- реализовать минимальную portfolio-admission / capital-governance semantics поверх уже существующих typed truths;
- встроить runtime в текущую runtime discipline без hidden bootstrap;
- не превращать фазу в protection line, `OMS` или broad workflow manager platform.

Главный принцип фазы:

> `P_14` строит foundation для узкого portfolio-governor / capital-governance contour, а не
> emergency supervision, order-management platform или broad manager workflow.

## 🧱 АРХИТЕКТУРНЫЕ ГРАНИЦЫ

`P_14` должна сохранить уже установленную ownership discipline.

### 1. Portfolio governor — это consumer existing typed truth

Новый слой не должен напрямую владеть:

- raw market data;
- analysis / intelligence truth;
- signal truth;
- base strategy truth;
- order registry ownership;
- emergency control ownership.

Он должен потреблять уже собранные typed truths через явный boundary.

### 2. Capital governance не должна подменять protection line

Даже если portfolio governor влияет на exposure и admission decisions, `P_14`
не должна автоматически превращаться в:

- kill switch;
- emergency liquidation engine;
- supervisor / freeze-all platform.

### 3. Portfolio governor не должна подменять OMS

`P_14` не владеет:

- order registry;
- cancel / modify lifecycle;
- reconciliation;
- exchange-facing order lifecycle semantics.

### 4. Runtime-time truth живёт в runtime

Dataclass/contracts layer должна держать только structural invariants.  
Runtime должна отдельно владеть lifecycle semantics и runtime-time truth.

---

## 🔌 RUNTIME / BOUNDARY DISCIPLINE

Фаза должна сохранить установленную runtime discipline проекта:

- explicit runtime entrypoint;
- явный `start()` / `stop()`;
- отсутствие hidden bootstrap;
- query/state-first surface;
- operator-visible diagnostics;
- явная degraded/readiness truth.

Composition root не должен:

- собирать доменный portfolio-governor context;
- выводить governor lifecycle-state;
- брать на себя ownership capital-governance semantics.

Bootstrap должен только:

- собрать runtime;
- wiring-ить upstream typed truth;
- встроить diagnostics в общую runtime/health truth.

## 🧩 ПРЕДВАРИТЕЛЬНАЯ ФОРМА RUNTIME CONTOUR

- как будет выглядеть query/state surface;
- как будут выражаться capital-governance / portfolio-admission semantics.

Ожидаемый характер contour:

- один narrow deterministic path;
- без emergency supervisor behavior;
- без order-management semantics;
- без broad workflow manager behavior.
