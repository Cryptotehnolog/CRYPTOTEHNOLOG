## 🎯 СМЫСЛ ФАЗЫ

`P_15` открывает первую отдельную линию `Protection / Supervisor Foundation`
поверх уже завершённых:

- `Execution Foundation`
- `Opportunity / Selection Foundation`
- `Strategy Orchestration / Meta Layer`
- `Position Expansion Foundation`
- `Portfolio Governor / Capital Governance Foundation`
- active `Risk Engine` / runtime discipline

Фактическая задача фазы:

- ввести `protection / supervisor` как самостоятельный production-compatible consumer contour;
- зафиксировать typed protection contracts и explicit runtime boundary;
- реализовать минимальную supervisory semantics поверх already existing typed truths;
- встроить protection runtime в текущую runtime discipline без hidden bootstrap;
- не превращать фазу в `OMS`, broad liquidation engine, notifications platform или broader workflow manager.

Главный принцип фазы:

> `P_15` строит foundation для узкого protection / supervisor contour, а не
> full emergency-operations platform, order-management platform или broad manager workflow.

## 🧱 АРХИТЕКТУРНЫЕ ГРАНИЦЫ

`P_15` должна сохранить уже установленную ownership discipline.

### 1. Protection / supervisor — это consumer existing typed truth

Новый слой не должен напрямую владеть:

- raw market data;
- analysis / intelligence truth;
- signal truth;
- strategy / opportunity / orchestration truth;
- order registry ownership;
- multi-channel notification ownership.

Он должен потреблять already existing typed truths через явный boundary.

### 2. Protection line не должна подменять OMS

`P_15` не владеет:

- order registry;
- cancel / modify lifecycle;
- reconciliation;
- exchange-facing order lifecycle semantics.

### 3. Protection line не должна подменять full emergency-operations platform

Даже если protection contour принимает supervisory decisions, `P_15`
не должна автоматически превращаться в:

- full liquidation engine;
- operator approval platform;
- incident-escalation platform;
- notifications / alert-delivery subsystem.

### 4. Runtime-time truth живёт в runtime

Dataclass/contracts layer должна держать только structural invariants.  
Runtime должна отдельно владеть lifecycle semantics и runtime-time truth.

---

## 🔌 RUNTIME / BOUNDARY DISCIPLINE

Фаза должна сохранить установленную runtime discipline проекта:

- explicit runtime entrypoint;
- явный `start()` / `stop()`;
- отсутствие hidden bootstrap;
- query/state-first surface;
- operator-visible diagnostics;
- явная degraded/readiness truth.

Composition root не должен:

- собирать protection domain context;
- выводить protection lifecycle-state вручную;
- брать на себя ownership supervisory semantics.

Bootstrap должен только:

- собрать protection runtime;
- wiring-ить upstream typed truth;
- встроить diagnostics в общую runtime/health truth.

## 🧩 ПРЕДВАРИТЕЛЬНАЯ ФОРМА RUNTIME CONTOUR

- без `OMS` ownership;
