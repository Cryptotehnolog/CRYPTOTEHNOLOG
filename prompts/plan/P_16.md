## 🎯 СМЫСЛ ФАЗЫ

`P_16` открывает первую отдельную линию `OMS Foundation` поверх уже завершённых:

- `Execution Foundation`
- `Opportunity / Selection Foundation`
- `Strategy Orchestration / Meta Layer`
- `Position Expansion Foundation`
- `Portfolio Governor / Capital Governance Foundation`
- `Protection / Supervisor Foundation`
- active `Risk Engine` / runtime discipline

Фактическая задача фазы:

- ввести `OMS` как самостоятельный production-compatible consumer contour;
- зафиксировать typed order-lifecycle contracts и explicit OMS runtime boundary;
- реализовать minimal centralized order-state / order-registry truth поверх existing execution truth;
- встроить OMS runtime в текущую runtime discipline без hidden bootstrap;
- не превращать фазу в smart-routing platform, liquidation engine, notifications / approval workflow или broader manager platform.

Главный принцип фазы:

> `P_16` строит foundation для узкого order-state / order-lifecycle contour, а не
> full trading-operations platform, emergency-ops platform или broad execution expansion.

## 🧱 АРХИТЕКТУРНЫЕ ГРАНИЦЫ

`P_16` должна сохранить уже установленную ownership discipline.

### 1. Execution владеет execution request / order-intent truth

Execution layer:

- потребляет strategy truth;
- строит `ExecutionContext`;
- формирует narrow execution intent truth;
- не становится centralized order registry.

`P_16` не должна переписывать execution semantics задним числом.

### 2. OMS владеет centralized order-state / lifecycle truth

OMS layer:

- потребляет existing execution truth;
- строит centralized order-state;
- владеет lifecycle semantics ордера;
- не подменяет execution runtime.

OMS должна быть system-of-record для narrow order-state truth, а не для всей trading-operations platform.

### 3. Portfolio Governor и Protection не получают order lifecycle ownership

`Portfolio Governor`:

- владеет capital-governance / portfolio-admission truth;
- не владеет order registry или order lifecycle.

`Protection`:

- владеет narrow supervisory truth;
- не владеет cancel / modify lifecycle;
- не владеет reconciliation;
- не владеет liquidation / notifications / approval workflow.

### 4. Future ops/notifications lines не должны маскироваться под OMS

Даже если OMS later может стать upstream source для future operational tooling,
в рамках `P_16` она не должна автоматически превращаться в:

- emergency-ops platform;
- incident remediation platform;
- notification delivery subsystem;
- approval / escalation platform.

---

## 🔌 RUNTIME / BOUNDARY DISCIPLINE

Фаза должна сохранить установленную runtime discipline проекта:

- explicit runtime entrypoint;
- явный `start()` / `stop()`;
- отсутствие hidden bootstrap;
- query/state-first surface;
- operator-visible diagnostics;
- явная degraded/readiness truth.

Composition root не должен:

- собирать OMS domain context;
- вручную моделировать lifecycle semantics ордера;
- брать на себя ownership order-state truth.

Bootstrap должен только:

- собрать OMS runtime;
- wiring-ить upstream execution truth;
- встроить diagnostics в общую runtime/health truth.

## 🧩 ПРЕДВАРИТЕЛЬНАЯ ФОРМА RUNTIME CONTOUR

До начала implementation-step уже должно быть концептуально ясно:

- какой runtime entrypoint будет у OMS layer;
- какой typed upstream truth будет её основным входом;
- как будет выглядеть query/state surface для active / historical orders;
- как будет выражаться lifecycle truth для narrow OMS contour.

Ожидаемый характер contour:

- один narrow deterministic ingest path;
- centralized order-state как operator-visible truth;
- без liquidation ownership;
- без notifications / approval ownership;
- без broader manager behavior;
- без превращения execution layer в implicit OMS.
