## 🎯 СМЫСЛ ЭТАПА

Фаза `P_8` открывает и закрывает первую отдельную signal-line поверх уже собранных:

- raw `market_data` truth;
- shared `analysis` truth;
- DERYA-first `intelligence` truth;
- действующей runtime / health / bootstrap discipline.

Её задача:

- ввести signal layer как самостоятельный production-compatible consumer contour;
- построить typed signal contracts и explicit signal runtime boundary;
- не превращать эту фазу в скрытую strategy platform, opportunity engine или supervisor line.

Главный принцип фазы:

> `P_8` должна построить честный foundation для signal generation, а не имитировать уже готовую multi-strategy торговую платформу.

## ✅ ЧТО ЕСТЬ НА СТАРТЕ P_8

После завершения `P_7` как `v1.7.0` в проекте уже есть:

- единый production bootstrap через composition root;
- operator-visible runtime truth для version / config / readiness / health / shutdown;
- один active production risk contour;
- raw `Market Data Layer` с typed contracts и `BAR_COMPLETED` truth;
- `IntelligenceRuntime` и `DeryaEngine` как DERYA-first intelligence foundation;
- `SharedAnalysisRuntime` в `src/cryptotechnolog/analysis` как source of truth для `ATR/ADX`;
- честно восстановленный production-compatible path до `RISK_BAR_COMPLETED`.

Фактический кодовый зазор перед `P_8`:

- production signal layer как отдельный runtime contour отсутствует;
- production `StrategyManager` отсутствует;
- signal contracts и signal event vocabulary не нормализованы;
- нет единого typed signal context contract поверх `market_data + analysis + intelligence`.

### 2. Prompt использует чужой package layout
В prompt используется структура вида:

- `src/signals/...`

Для `P_8` принимается:

- новый код размещается внутри текущей package discipline проекта;
- phase plan не должен навязывать отдельный верхнеуровневый `src/signals`, пока это явно не согласовано архитектурно.

### 3. Prompt смешивает несколько будущих фаз в одну
В prompt одновременно присутствуют:

- `SignalGenerator`;
- `OpportunityEngine`;
- `MetaClassifier`;
- несколько стратегий;
- pyramiding;
- conflict resolution;
- persistence;
- ranking/supervisor semantics.

Для `P_8` принимается:

- Phase 8 открывается как `Signal Generation Foundation`;
- ranking / orchestration / supervisor logic остаются follow-up lines.

### 4. Prompt размывает ownership слоёв
В prompt signal layer местами предполагает, что:

- risk отдаёт indicator-like truth;
- intelligence already owns all signal-oriented semantics;
- market data, analysis, intelligence и risk можно читать как один общий источник.

Для `P_8` принимается:

- signal layer является consumer-ом существующих truth layers;
- он не владеет `market_data`, `analysis`, `intelligence` или `risk` truth.

### 5. Prompt преждевременно требует persistence schema
В prompt persistence tables и audit contours вводятся до contract lock.

Для `P_8` принимается:

- signal persistence не является обязательным first-pass scope;
- сначала фиксируются signal contracts, runtime boundary и operator truth.

## 🧱 ЦЕЛЬ ФАЗЫ

Построить минимальный production-compatible signal layer, который:

1. потребляет уже существующую truth из:
   - raw market data;
   - shared analysis;
   - intelligence;
2. собирает signal context детерминированно и без hidden bootstrap;
3. формирует typed signal outputs;
4. имеет explicit runtime boundary;

6. остаётся отделённым от strategy orchestration, opportunity ranking и portfolio-level решений.

## 🔒 ОБЯЗАТЕЛЬНЫЕ ИНЖЕНЕРНЫЕ ПРИНЦИПЫ ФАЗЫ

### 1. Signal layer является consumer contour
Signal layer потребляет:

- raw `market_data` truth;
- shared `analysis` truth;
- `intelligence` truth.

Он не становится владельцем этих слоёв.

### 2. Contract-first важнее ширины feature scope
Сначала фиксируются:

- signal contracts;
- signal validity / readiness semantics;
- signal runtime boundary;
- signal event vocabulary;

и только потом расширяется signal logic.

### 3. Event truth не заменяет state/query truth
Signal events допустимы, но signal layer должен иметь:

- state/query-first surface;
- operator-visible runtime truth;
- deterministic signal context assembly.

### 4. Никакого bootstrap хаоса
Signal runtime входит только через composition root:

- без import-time subscriptions;
- без global singleton shortcuts;
- без скрытых background workers.

### 5. Operator truth должна отражать правду
Signal layer должен показывать:

- started / ready / degraded state;
- warming / missing-input semantics;
- причины недоступности signal generation;
- текущее качество signal context assembly.

## 🧾 DEFINITION OF DONE

Фаза `P_8` считается завершённой только если одновременно выполнено всё ниже:

1. У платформы есть production-compatible `Signal Generation Foundation`.
2. Signal layer выражен через typed contracts и signal context contract.
3. Signal runtime встроен в runtime discipline, унаследованную из `P_5_1`, `P_6`, `P_7` и `C_7R`.
4. Signal layer потребляет существующие truth layers как consumer contour и не размывает ownership между слоями.
5. Есть один узкий, детерминированный signal generation contour.
6. Signal runtime имеет readiness / health / degradation / operator truth.
7. Фаза не затянула внутрь себя opportunity ranking, meta classification, multi-strategy orchestration, pyramiding или dashboard line.
8. Unit и integration tests проверяют реальный runtime contract фазы.
9. Документация и phase-plan отражают фактический scope, а не старую раздутость исторического prompt.
