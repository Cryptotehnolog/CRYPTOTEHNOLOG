## 🎯 СМЫСЛ ЭТАПА

Фаза `P_7` открывает следующую основную feature-line после завершённой `P_6`.

Её задача:
- построить production-compatible слой индикаторов и intelligence-оценок поверх уже введённого `Market Data Layer`;
- сделать его единым analysis foundation для следующих фаз strategy/signal line;
- не смешать этот слой с преждевременной генерацией сигналов, opportunity ranking или dashboard track.

Главный принцип фазы:

> `P_7` должна построить вычислительный и semantic foundation для анализа рынка, а не имитировать уже готовый Signal Generator или Strategy Manager.

## ✅ ЧТО ЕСТЬ НА СТАРТЕ P_7

После завершения `P_5_1` и `P_6` в проекте уже есть:

- единый production bootstrap через composition root;
- runtime truth для version / config / bootstrap / health / shutdown;
- один active production risk path;
- event-driven foundation и lifecycle discipline;
- завершённый `Market Data Layer` в `src/cryptotechnolog/market_data`;
- typed contracts для tick / bar / orderbook / symbol metrics / universe;
- readiness / degraded / blocked semantics для market data;
- готовый `Risk Engine`, который уже может быть потребителем future intelligence signals.

Фактический кодовый зазор перед `P_7`:

- `src/cryptotechnolog/intelligence` существует как package boundary, но ещё не наполнен production-ready foundation;
- `src/cryptotechnolog/strategy` и `src/cryptotechnolog/execution` ещё не являются полноценной следующей runtime-line;
- dashboard line существует отдельно и не должна определять scope основной фазовой ветки.

## ❗ ЧТО НУЖНО СРАЗУ ПЕРЕСМОТРЕТЬ В ИСХОДНОМ PROMPT ФАЗЫ 7

Исходный файл:
[08_ФАЗА_7_INDICATORS_PROMPT.md](/D:/CRYPTOTEHNOLOG/prompts/08_%D0%A4%D0%90%D0%97%D0%90_7_INDICATORS_PROMPT.md)

Он силён по замыслу, но в текущем виде содержит scope inflation и архитектурную двусмысленность.

### 1. Неверный package layout
В prompt используется структура вида:
- `src/indicators/...`
- `src/intelligence/...`
- прямые ссылки на layout, не совпадающий с текущим package path проекта.

Для `P_7` принимается:
- весь новый Python-код размещается внутри:
  - `src/cryptotechnolog/intelligence`
  - при необходимости `src/cryptotechnolog/analysis`
- нельзя вводить новый верхнеуровневый `src/indicators` вне текущей package structure.

### 2. Scope prompt слишком широк
Prompt одновременно пытается включить:
- indicator library;
- market regime logic;
- liquidity intelligence;
- stop hunt semantics;
- MAE/MFE tracking;
- Donchian breakout semantics;
- части future signal/strategy line.

Для `P_7` принимается:
- Phase 7 строит foundation contracts/runtime для indicators + intelligence;
- Phase 7 не считается phase closure для Strategy Manager, Signal Generator, OpportunityEngine или MetaClassifier.

### 3. Prompt опирается на неточную truth-картину P_6
В prompt встречается язык, как будто в `P_6` уже полностью реализованы:
- live websocket manager как основная truth;
- storage/runtime history path;
- широкий intelligence-ready ingestion beyond current foundation.

Для `P_7` принимается:
- `P_6` завершила contract-first market data foundation;
- `P_7` строится поверх фактического `MarketDataRuntime`, а не поверх выдуманного legacy manager-style scope.

### 4. Смешение indicators и signal-generation semantics
Prompt местами оформляет indicator/intelligence слой так, как будто он уже должен:
- генерировать trading signals;
- решать strategy conflicts;
- ранжировать торговые возможности;
- задавать исполнение сделок.

Для `P_7` принимается:
- intelligence layer может публиковать typed analysis events;
- но он не отвечает за торговое решение как таковое;
- signal-generation остаётся следующей отдельной phase-line.

### 5. DERYA нельзя оформлять как “магический микроструктурный движок”
DERYA в рамках `P_7` допустим и полезен, но только при честной инженерной формализации.

Для `P_7` принимается:
- DERYA использует `OHLCV bar-efficiency proxy`, а не истинную tick-level microstructure;
- DERYA должен быть оформлен как отдельный intelligence module;
- его режимы допускаются только как deterministic regime machine с жёсткими правилами, а не как расплывчатые qualitative labels.

### 6. Scope truth после текущей DERYA-first реализации линии
После выполнения текущих шагов `P_7` принимается следующая нормализация phase scope:
- explicit runtime entrypoint текущей линии: `IntelligenceRuntime`;
- фактически реализованный first-class intelligence factor текущей фазы: `DERYA` через `DeryaEngine` и typed query/event contracts;
- corrective line `C_7R` дополняет release truth отдельным shared analysis layer в `src/cryptotechnolog/analysis`, где живут derived inputs `ATR/ADX` для active risk contour;
- corrective line `C_7R` восстанавливает production-compatible path до `RISK_BAR_COMPLETED`, не смешивая raw market-data `BAR_COMPLETED` с risk-specific payload;
- typed indicator contracts зафиксированы, но широкая classical indicator library/runtime не считается частью closure текущей DERYA-first линии, если она не подтверждена фактическим кодом;
- config hot-reload для thresholds/periods и более широкий indicator runtime остаются follow-up line и не должны приписываться фазе задним числом;
- observability текущего closure scope означает readiness/degraded/operator diagnostics и DERYA transition/query truth, а не dashboard-level regime distribution.

## 🧱 ЦЕЛЬ ФАЗЫ

Построить production-ready `Indicators + Intelligence Foundation`, который:

1. принимает bar/tick/orderbook inputs из существующего `Market Data Layer`;
2. рассчитывает typed indicator values без bootstrap side effects;
3. формирует typed intelligence assessments поверх market data contracts;
4. вводит explicit runtime foundation текущей фазы через `IntelligenceRuntime`, а для corrective recovery использует `SharedAnalysisRuntime` как source/truth layer derived inputs;
5. делает `DERYA` частью intelligence foundation как bar-efficiency regime model;
6. публикует analysis-level events и operator-visible diagnostics;
7. оставляет strategy/signal line следующей отдельной фазой, а не прячет её внутрь текущего этапа.

## 🔒 ОБЯЗАТЕЛЬНЫЕ ИНЖЕНЕРНЫЕ ПРИНЦИПЫ ФАЗЫ

### 1. Intelligence layer не равен signal layer
Indicators и intelligence выдают analysis truth, но не принимают торговое решение за систему.

### 2. Contract-first важнее ширины scope
Сначала фиксируются typed models, events, runtime boundaries и semantics, и только потом добавляется вычислительная ширина.

### 3. DERYA должен быть честным по входам и смыслу
DERYA в этой фазе — это `bar-efficiency regime analyzer`, а не tick/orderbook microstructure engine.

### 4. Regime semantics должны быть детерминированными
`Expansion / Exhaustion / Collapse / Recovery` допускаются только как формально определённая regime machine:
- через пороги;
- через сглаживание;
- через directional slope rules;
- через anti-flap policy.

### 5. Никакого нового bootstrap хаоса
Новый analysis/intelligence runtime входит в существующий composition-root discipline и не создаёт import-time side effects.

### 6. Observability должна отражать правду
Operator-facing diagnostics должны показывать:
- готовность indicator/intelligence runtime;
- coverage по символам/таймфреймам;
- DERYA regime distribution;
- деградации из-за нехватки данных или плохого quality input.

## 🧾 DEFINITION OF DONE

Фаза `P_7` считается завершённой только если одновременно выполнено всё ниже:

1. У платформы есть production-compatible `Indicators + Intelligence Foundation`.
2. Indicator values и intelligence assessments выражены через typed contracts.
3. Intelligence layer встроен в runtime discipline, введённую в `P_5_1` и `P_6`.
4. DERYA реализован как отдельный intelligence factor с формально определённой 4-режимной machine.
5. DERYA честно оформлен как `OHLCV bar-efficiency proxy`, а не как ложная tick-level microstructure feature.
6. Intelligence runtime имеет readiness/health/degradation/operator truth.
7. Фаза не затянула внутрь себя signal generation, opportunity ranking, strategy management или dashboard line.
8. Unit и integration tests проверяют не только формулы, но и реальный runtime contract фазы.
9. Документация и phase-plan отражают фактический scope, а не исходную раздутость старого prompt.

Для closure текущей DERYA-first реализации это трактуется так:
- explicit runtime entrypoint текущей closure-реализации: `IntelligenceRuntime`;
- corrective runtime/source-of-truth line текущей closure-реализации: `SharedAnalysisRuntime` в `src/cryptotechnolog/analysis`;
- first-class intelligence factor текущей closure-реализации: `DERYA`;
- `ATR/ADX` и production-compatible `RISK_BAR_COMPLETED` входят в итоговую release truth только через corrective line `C_7R`, а не как задним числом приписанная broad indicator runtime;
- typed indicator contracts входят в foundation scope, но отсутствие широкой production indicator library само по себе не означает, что DERYA-first линия притворяется реализованной;
- config/hot-reload и расширенный indicator runtime считаются follow-up work, пока они не подтверждены кодом и отдельным runtime contract;
- operator truth текущего closure scope означает readiness/health/degradation diagnostics и typed DERYA query/event surface, а не dashboard-first визуализацию.

## 🚀 РЕЗУЛЬТАТ ФАЗЫ

После завершения `P_7` платформа должна перейти в состояние:

- не “market data есть, но analysis layer ещё расплывчатый и полу-концептуальный”;

Это создаёт правильную основу для следующего этапа:
- `P_8` как отдельной signal-generation / opportunity line;
- без смешения с dashboard track;
- и без фальшивого ощущения, что strategy/execution уже собраны внутри самой Phase 7.
