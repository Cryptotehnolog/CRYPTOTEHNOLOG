## 🧱 АРХИТЕКТУРНЫЕ ГРАНИЦЫ

Closure-ready `P_17` сохраняет уже установленную ownership discipline.

### 1. Manager координирует existing truths, а не подменяет их

Manager layer:

- потребляет existing typed truths;
- координирует workflow-level path;
- не переписывает задним числом ownership соседних runtime layers.

### 2. Manager не подменяет Execution и OMS

`Execution`:

- владеет `order-intent` truth;
- не становится manager submodule.

`OMS`:

- владеет `order-state / lifecycle` truth;
- не становится manager-owned registry.

Manager может читать downstream OMS truth, но не получает ownership над order lifecycle.

### 3. Manager не подменяет Opportunity / Orchestration

`Opportunity`:

- владеет selection truth;
- не должна автоматически считаться частью broader manager logic.

`Orchestration`:

- владеет narrow meta-decision / `FORWARD` / `ABSTAIN` semantics;
- не должна автоматически объявляться full workflow manager.

`P_17` строится поверх них, а не вместо них.

### 4. Manager не подменяет Position Expansion / Portfolio Governor / Protection

`Position Expansion`:

- владеет add-to-position contour.

`Portfolio Governor`:

- владеет capital-admission truth.

`Protection`:

- владеет narrow supervisory truth.

Manager может координировать runtime-level workflow поверх этих truths,
но не получает ownership над:

- capital governance;
- supervisory decisions;
- liquidation / approval / ops semantics.

### 5. Future operational and validation lines не должны маскироваться под Manager

Даже если future manager line later может быть upstream source для broader workflow tooling,
в рамках `P_17` она не должна автоматически превращаться в:

- notifications / approval workflow platform;
- liquidation / emergency-ops platform;
- analytics / validation platform;
- dashboard-led operations surface.

## 🔌 RUNTIME / BOUNDARY DISCIPLINE

Closure-ready реализация сохраняет установленную runtime discipline проекта:

- explicit runtime entrypoint;
- явный `start()` / `stop()`;
- отсутствие hidden bootstrap;
- query/state-first surface;
- operator-visible diagnostics;
- явная degraded/readiness truth.

Composition root не должен:

- собирать manager domain context;
- вручную моделировать manager workflow semantics;
- брать на себя ownership соседних layer truths.

Bootstrap должен только:

- собрать manager runtime;
- wiring-ить upstream typed truths;
- встроить diagnostics в общую runtime/health truth.
