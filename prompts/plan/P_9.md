## 🎯 СМЫСЛ ФАЗЫ

`P_9` вводит первую отдельную strategy-line поверх уже завершённой `Signal Generation Foundation`.

Фактическая задача фазы:

- ввести strategy layer как самостоятельный production-compatible consumer contour;
- зафиксировать typed strategy contracts и explicit strategy runtime boundary;
- реализовать минимальную strategy-level action candidate semantics поверх signal truth;
- встроить strategy runtime в существующую runtime discipline без hidden bootstrap;
- не превращать фазу в `Portfolio Governor`, supervisor line или скрытую multi-strategy platform.

Главный принцип фазы:

> `P_9` строит foundation для strategy-level consumption signal truth, а не portfolio/runtime platform.

## ✅ ЧТО РЕАЛЬНО РЕАЛИЗОВАНО

### 1. Package boundary

Реализован отдельный package boundary:

- `src/cryptotechnolog/strategy`

Strategy layer встроена в текущую package discipline проекта и не уводится в внешний `src/strategy` или `src/portfolio`.

### 2. Strategy contracts

Зафиксированы typed strategy contracts:

- `StrategyDirection`
- `StrategyStatus`
- `StrategyValidityStatus`
- `StrategyReasonCode`
- `StrategyValidity`
- `StrategyFreshness`
- `StrategyContext`
- `StrategyActionCandidate`

Что это даёт:

- strategy layer является consumer-ом signal truth;
- strategy layer не живёт на сырых dict;
- structural invariants отделены от runtime-time truth;
- strategy candidate semantics выражена отдельно от signal truth.

### 3. Strategy event vocabulary

Зафиксирован узкий typed strategy event vocabulary:

- `STRATEGY_CANDIDATE_UPDATED`
- `STRATEGY_ACTIONABLE`
- `STRATEGY_INVALIDATED`

Event vocabulary остаётся narrow:

- без portfolio semantics;
- без opportunity/meta universe;
- без execution lifecycle.

### 4. Explicit StrategyRuntime

Реализован explicit `StrategyRuntime`, который:

- стартует и останавливается явно;
- не делает hidden bootstrap;
- принимает signal truth через явный ingest boundary;
- сам собирает `StrategyContext` внутри strategy layer;
- хранит query/state-first truth;
- публикует operator-visible diagnostics;
- сам решает lifecycle/freshness transitions на runtime-уровне.

### 5. Minimal strategy contour

Реализован один узкий deterministic strategy contour:

- strategy layer потребляет `SignalSnapshot`, а не upstream `market_data` / `analysis` / `intelligence` напрямую;
- actionable `BUY` signal даёт `LONG` strategy candidate;
- actionable `SELL` signal даёт `SHORT` strategy candidate;
- non-actionable / warming / invalid signal states не маскируются под ложный actionable output.

### 6. Narrow runtime integration

`StrategyRuntime` встроен в composition root:

- strategy runtime собирается в `bootstrap`;
- diagnostics strategy layer включены в platform health/runtime truth;
- strategy ingest идёт от signal event boundary;
- bootstrap не собирает `StrategyContext`;
- composition root только проводит signal truth в strategy runtime.

### 7. Operator-visible truth

Для strategy layer реализованы:

- `started / ready / degraded` semantics;
- tracked / actionable / invalidated / expired candidate counts;
- last failure reason;
- readiness reasons;
- degraded reasons;
- shutdown cleanup truth.

## 🧭 АРХИТЕКТУРНЫЕ ГРАНИЦЫ

### 1. Strategy layer является consumer-ом signal truth

Strategy layer:

- потребляет `SignalSnapshot`;
- не владеет `market_data`, `analysis`, `intelligence`, `signals` или `risk` truth;
- не подменяет execution layer;
- не подменяет portfolio governance.

### 2. Bootstrap не владеет strategy semantics

Composition root:

- wiring-ит existing runtime contours;
- передаёт signal truth в `StrategyRuntime`;
- не собирает `StrategyContext`;
- не реализует strategy decision logic.

### 3. Event truth не заменяет state/query truth

Strategy layer публикует typed events, но основная truth остаётся:

- query/state-first;
- operator-visible;
- deterministic.

### 4. Runtime-time truth живёт в runtime

Expiry / invalidation / degradation semantics определяются в runtime, а не подменяются dataclass-level псевдо-temporal логикой.
