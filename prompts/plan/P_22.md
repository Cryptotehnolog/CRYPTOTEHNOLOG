# P_22 - Live Feed Connectivity Foundation

**Статус:** Phase opening truth assembled  
**Дата:** 2026-03-24

## 1. Смысл фазы

`P_22` открывает `Live Feed Connectivity Foundation` как узкую
live-feed/session connectivity line.

Эта линия нужна не для построения широкой exchange-connectivity platform,
не для adapter ecosystem
и не для execution/routing/reconciliation stack,
а для формализации typed live feed/session truth поверх уже существующего
`market_data` foundation contour.

`P_22` на opening-step не является:

- broad exchange adapter platform;
- execution connectivity line;
- routing / smart-routing line;
- failover / reliability platform;
- reconciliation platform;
- historical storage / backfill line;
- dashboard / operator workflow line.

## 2. Стартовая truth после `v1.21.0`

После `v1.21.0` и formal closure `P_21` проект уже имеет:

- `market_data` как typed domain/runtime foundation;
- `execution` как отдельный execution contour;
- `oms` как отдельный order-lifecycle/order-state contour;
- README / phase docs / ADR truth, которые честно выносят:
  - live websocket/feed connectivity;
  - exchange adapters;
  - routing / reconciliation / failover
  за пределы уже закрытых фаз.

Это позволяет открыть не broad exchange-connectivity territory,
а более узкий и честный future contour:

- `Live Feed Connectivity Foundation`.

Opening truth этой фазы:

- `P_22 = Live Feed Connectivity Foundation`
- это узкая live-feed/session connectivity line;
- она владеет typed connection/session truth и narrow ingest handoff;
- она не владеет `market_data` domain contracts целиком;
- она не владеет `execution`, `oms`, routing или reconciliation semantics.

## 3. In Scope

В scope `P_22` на opening-step входят только:

- отдельный package boundary для live feed connectivity layer;
- typed connection status/state truth;
- typed session identity truth;
- typed feed-health/readiness/degraded truth;
- typed ingress envelope / ingest handoff;
- minimal reconnect/backoff semantics;
- narrow connectivity runtime boundary;
- narrow live-ingest path в existing `market_data` layer;
- unit/integration verification на relevant connectivity subset.

## 4. Exact Ownership Boundary

`Live Feed Connectivity Foundation` владеет только:

- connection/session lifecycle truth;
- transport-facing connectivity state;
- feed-health/readiness/degraded semantics;
- narrow typed ingest handoff в `market_data`;
- minimal reconnect/backoff semantics;
- connectivity runtime boundary.

`Live Feed Connectivity Foundation` не владеет:

- `market_data` domain contracts целиком;
- tick / bar / orderbook semantics;
- symbol metrics / universe semantics;
- `execution`;
- `oms`;
- routing / smart-routing semantics;
- reconciliation semantics;
- delivery/operator workflow semantics.

Это означает:

- `market_data` остаётся owner-ом market-data domain truth;
- `execution` остаётся owner-ом execution truth;
- `oms` остаётся owner-ом order-state / order-lifecycle truth;
- `connectivity` только формирует live session/connectivity truth
  и узкий ingest boundary.

## 5. Exact Typed Core

Минимальный honest typed core для opening-step:

- `FeedConnectionStatus`
- `FeedConnectionState`
- `FeedSessionIdentity`
- `FeedConnectivityAssessment`
- `FeedIngressEnvelope`
- narrow ingest contract

### 5.1 `FeedConnectionStatus`

Минимальный lifecycle status должен честно различать:

- `DISCONNECTED`
- `CONNECTING`
- `CONNECTED`
- `DEGRADED`

### 5.2 `FeedSessionIdentity`

Session identity должна минимально фиксировать:

- `exchange`
- `stream_kind`
- `subscription scope`

### 5.3 `FeedConnectionState`

Connection state должна минимально удерживать:

- `session_id`
- `status`
- `connected_at`
- `last_message_at`
- `degraded_reason`, если применимо
- minimal state revision / sequence, если это нужно для deterministic truth

### 5.4 `FeedConnectivityAssessment`

Assessment должен отдельно формализовать:

- ready / degraded / blocked truth;
- degraded reason;
- stale/lag marker, если он уже нужен для narrow feed-health semantics.

### 5.5 `FeedIngressEnvelope`

Ingress envelope должен быть typed handoff между connectivity layer и
existing `market_data` layer:

- session identity;
- ingest timestamp;
- payload kind;
- raw transport payload reference или minimal normalized transport payload.

## 6. Exact Boundary Between Session Truth and `market_data`

### Connectivity layer

Владеет:

- connection/session lifecycle;
- connectivity health/degraded truth;
- narrow ingest handoff.

### `market_data`

Владеет:

- tick/bar/orderbook models;
- data quality semantics;
- symbol metrics;
- universe semantics.

Следовательно:

- connectivity layer не подменяет `MarketDataRuntime`;
- connectivity layer не становится owner-ом market data domain models;
- `market_data` принимает ingress, но не обязана владеть session/connectivity truth.

## 7. Почему package boundary нужна уже на opening

Package boundary нужна уже на opening-step, потому что без неё future connectivity
truth начнёт сразу расползаться по:

- `market_data/runtime`;
- future adapter/client logic;
- execution-adjacent connectivity semantics;
- broader exchange integration expectations.

Отдельный package boundary нужен как ownership lock:

- connectivity line владеет session/connectivity truth;
- `market_data` сохраняет ownership над domain contracts;
- будущая line не маскируется под “небольшое расширение” уже закрытого `market_data`.

## 8. Почему узкая runtime boundary нужна уже на opening

В отличие от artifact-only phase, эта линия по смыслу уже lifecycle-oriented.

Поэтому на opening-step нужна узкая runtime boundary:

- explicit start/stop path;
- session state transitions;
- ready/degraded/disconnected truth;
- minimal reconnect/backoff behavior.

Но эта runtime boundary не должна становиться:

- service platform;
- orchestration platform;
- API/delivery layer;
- broad exchange integration system.

## 9. Почему adapter ecosystem на opening не нужна

На opening-step не нужен:

- broad exchange adapter ecosystem;
- rich client hierarchy;
- multi-exchange connector platform.

Допустим только very narrow transport/client contract,
если он реально нужен для honest live-feed connectivity truth.

Причина:

- adapter ecosystem слишком быстро тянет за собой:
  - multi-exchange abstraction;
  - execution connectivity;
  - routing;
  - failover/reliability platform;
  - broader integration ownership.

## 10. Что допустимо по reconnect/backoff semantics

На opening-step допустимы только:

- minimal reconnect intent / retry marker;
- `retry_count`;
- `next_retry_at`, если нужен;
- `last_disconnect_reason`;
- deterministic degraded transition при stale feed / disconnect / repeated retry failure.

На opening-step не допустимы:

- rich reconnection strategy hierarchy;
- adaptive multi-venue failover;
- connector pool orchestration;
- advanced recovery workflows;
- reliability platform policy engine.

## 11. Out Of Scope

На opening-step жёстко вне scope:

- broad exchange adapter ecosystem;
- rich client hierarchy;
- execution connectivity;
- order routing;
- smart routing;
- failover / reliability platform;
- reconciliation;
- historical storage / backfill platform;
- dashboard/operator workflows;
- analytics / reporting / research semantics;
- `market_data` domain ownership takeover;
- `execution` ownership;
- `oms` ownership;
- broad live-trading platform rewrite.

## 12. Acceptance Criteria / Definition of Done

Фаза может считаться closure-ready только если:

- существует отдельный package boundary для connectivity layer;
- typed connection/session contracts формально определены;
- typed feed-health/readiness/degraded truth существует отдельно от `market_data` domain truth;
- typed ingress envelope / ingest contract существует явно;
- введён narrow connectivity runtime boundary;
- minimal reconnect/backoff semantics удерживается узко и честно;
- `market_data` сохраняет ownership над domain contracts;
- opening implementation не вводит adapter ecosystem, execution connectivity,
  routing, reconciliation или reliability platform semantics;
- README, plan, result doc и final ADR не противоречат этой узкой opening truth.

## 13. Anti-Scope Guards

В рамках `P_22` нельзя честно делать вид, что live-feed connectivity line уже
является:

- exchange adapter platform;
- execution connectivity layer;
- routing / smart-routing line;
- failover / reliability platform;
- reconciliation platform;
- historical data platform;
- dashboard/operator line.

Любой шаг, который реально требует этих semantics,
должен открываться отдельно,
а не маскироваться под normal continuation `P_22`.

## 14. Note On Historical / Reference Materials

Historical broad exchange-connectivity expectations используются только как
reference-only context.

Implementation truth для `P_22` задаётся только:

- current plan truth;
- current ADR boundaries;
- current README truth;
- current code truth existing adjacent layers.

## 15. Note On ADR Before Finalization

На opening-step отдельный ADR для `P_22` пока не обязателен.

Честная последовательность для этой линии:

1. сначала открыть `P_22` как узкую `Live Feed Connectivity Foundation`;
2. затем, если line реально идёт к finalization, зафиксировать отдельный ADR
   до formal finalization;
3. не позволять interpretation drift расширить линию в adapter/routing/
   reliability/reconciliation semantics.
