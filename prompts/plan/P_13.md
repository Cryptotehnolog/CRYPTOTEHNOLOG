## 🎯 СМЫСЛ ФАЗЫ

`P_13` открывает первую отдельную линию `Position Expansion Foundation`
поверх уже завершённых:

- `Strategy Foundation`
- `Execution Foundation`
- `Opportunity / Selection Foundation`
- `Strategy Orchestration / Meta Layer`
- active `Risk Engine` / runtime discipline

Фактическая задача фазы:

- ввести `position expansion` как самостоятельный production-compatible consumer contour;
- зафиксировать typed position-expansion contracts и explicit runtime boundary;
- реализовать минимальную add-to-position / pyramiding semantics поверх уже существующих typed truths;
- встроить runtime в текущую runtime discipline без hidden bootstrap;
- не превращать фазу в portfolio governor, protection line, OMS или broad manager platform.

Главный принцип фазы:

> `P_13` строит foundation для узкого add-to-position contour, а не
> portfolio-wide governance, emergency supervision или order-management platform.

## 🧱 АРХИТЕКТУРНЫЕ ГРАНИЦЫ

`P_13` должна сохранить уже установленную ownership discipline.

### 1. Position expansion — это consumer existing typed truth

Новый слой не должен напрямую владеть:

- raw market data;
- analysis / intelligence truth;
- signal truth;
- base strategy truth;
- execution submission ownership;
- portfolio-wide governance;
- emergency control ownership.

Он должен потреблять уже собранные typed truths через явный boundary.

### 2. Expansion semantics не должна подменять portfolio line

Даже если add-to-position влияет на exposure и risk profile, `P_13`
не должна автоматически превращаться в:

- capital allocation engine;
- portfolio-level exposure gate;
- supervisor / governor platform.

### 3. Expansion semantics не должна подменять OMS / protection line

`P_13` не владеет:

- order registry;
- cancel / modify lifecycle;
- reconciliation;
- emergency close-all / freeze / kill-switch semantics.

### 4. Runtime-time truth живёт в runtime

Dataclass/contracts layer должна держать только structural invariants.  
Runtime должна отдельно владеть lifecycle semantics и runtime-time truth.

---

## 🔌 RUNTIME / BOUNDARY DISCIPLINE

Фаза должна сохранить установленную runtime discipline проекта:

- explicit runtime entrypoint;
- явный `start()` / `stop()`;
- отсутствие hidden bootstrap;
- query/state-first surface;
- operator-visible diagnostics;
- явная degraded/readiness truth.

Composition root не должен:

- собирать доменный expansion context;
- выводить expansion lifecycle-state;
- брать на себя ownership add-to-position semantics.

Bootstrap должен только:

- собрать runtime;
- wiring-ить upstream typed truth;
- встроить diagnostics в общую runtime/health truth.

---

## 🧩 ПРЕДВАРИТЕЛЬНАЯ ФОРМА RUNTIME CONTOUR

До начала implementation-step уже должно быть концептуально ясно:

- какой runtime entrypoint будет у position-expansion layer;
- какой typed upstream truth будет её основным входом;
- как будет выглядеть query/state surface;
- как будут выражаться add-to-position eligibility и abstain / reject semantics.

Ожидаемый характер contour:

- один narrow deterministic path;
- без broad portfolio orchestration;
- без order-management semantics;
- без emergency supervisor behavior.
