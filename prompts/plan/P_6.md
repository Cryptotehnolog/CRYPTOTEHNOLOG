## 🎯 СМЫСЛ ЭТАПА

Фаза `P_6` открывает следующую основную линию проекта после production alignment.

Её задача:
- построить production-ready слой рыночных данных;
- сделать его единственным source of truth для tick / bar / orderbook / symbol liquidity signals;
- ввести `UniverseEngine` как управляемый контур допуска инструментов к торговле;
- не разрушить runtime discipline, введённую в `P_5_1`.

Главный принцип фазы:

> Market Data Layer не должен быть просто “поставщиком тиков”.  
> Это системный слой, который определяет качество данных, управляет admissible universe и влияет на готовность платформы к торговле.

---

## ✅ ЧТО ЕСТЬ НА СТАРТЕ P_6

После завершения `P_5_1` в проекте уже есть:

- единый production bootstrap через composition root;
- один active production risk path;
- runtime truth для version / config / bootstrap / active risk path;
- fail-fast / degraded startup semantics;
- health / readiness / lifecycle audit trail;
- event-driven foundation;
- Risk Engine с event vocabulary production-уровня.

Это означает:
- Фаза 6 может опираться на уже собранный runtime;
- Но не имеет права возвращать скрытые bootstrap-path, import-time side effects или ad-hoc wiring.

## ❗ ЧТО НУЖНО СРАЗУ ПЕРЕСМОТРЕТЬ В ИСХОДНОМ PROMPT ФАЗЫ 6

Исходный файл:
[07_ФАЗА_6_MARKET_DATA_PROMPT.md](/D:/CRYPTOTEHNOLOG/prompts/07_ФАЗА_6_MARKET_DATA_PROMPT.md)

Сильный по замыслу, но в текущем виде он требует уточнения перед реализацией.

### 1. Путь модулей
В prompt используется структура вида:
- `src/market_data/...`
- `src.core.logger`
- `python -m src.main`

Это не соответствует текущей архитектуре проекта после `P_5_1`.

Для `P_6` принимается:
- весь новый Python-код размещается в `[src/cryptotechnolog/market_data](/D:/CRYPTOTEHNOLOG/src/cryptotechnolog/market_data)`
- интеграции идут через существующие модули `cryptotechnolog.config`, `cryptotechnolog.core`, `cryptotechnolog.risk`, `cryptotechnolog.bootstrap`

### 2. Версионный язык prompt
В prompt встречается naming вроде:
- `Risk Engine v4.4`
- `Market Data Layer v4.4`

Для фазового плана проекта это создаёт ненужную двусмысленность.

Для `P_6` принимается:
- фазовая версия проекта: `v1.6.0`
- внутренние “v4.4” обозначения считаются историческими/описательными и не используются как release identity

### 3. Event publication contract
В prompt примеры публикации местами записаны как произвольные dict payloads.

После `P_5_1` это уже недостаточно строго.

Для `P_6` принимается:
- публикация событий идёт через существующий event contract проекта;
- event type, source, priority, payload и correlation semantics должны быть совместимы с `EnhancedEventBus`, audit и metrics listeners;
- нельзя вводить “второй стиль” event publication параллельно существующему.

### 4. UniverseEngine и state transitions
Prompt правильно хочет связать `UniverseEngine` со `State Machine`, но сейчас это сформулировано слишком прямолинейно.

Для `P_6` принимается:
- `UniverseEngine` не делает произвольные state transitions “сам по себе”;
- он публикует типизированные сигналы / вызывает явный orchestration path, совместимый с Control Plane;
- state changes остаются под контролем System Controller / State Machine contract.

### 5. Funding / ranked universe / future phases
Prompt затрагивает:
- funding-based filters,
- ranked universe,
- OpportunityEngine integration,
- Strategy Manager integration.

Для `P_6` принимается:
- raw + admissible universe обязательны;
- ranked universe можно закладывать как contract/storage/API, но не расползаться в полную реализацию OpportunityEngine;
- funding signals допустимы как часть `SymbolMetrics`, но не должны раздувать фазу в глубокую funding/opportunity feature line.

### 6. Scope truth после фактической реализации фазы
После выполнения шагов `P_6` принимается следующая нормализация phase scope:
- explicit runtime entrypoint текущей фазы: `MarketDataRuntime`;
- admissible/universe orchestration текущей фазы: `UniversePolicy` + `MarketDataRuntime.refresh_universe(...)`;
- отдельные `MarketDataManager` и `UniverseEngine` из исходного prompt считаются историческим manager-style language, а не обязательными именами классов для closure `P_6`;
- websocket/feed runtime и persistence/storage path не входят в обязательный closure scope текущей фазы и остаются future follow-up line.

## 🧱 ЦЕЛЬ ФАЗЫ

Построить production-ready `Market Data Layer`, который:

1. получает и валидирует рыночные данные;
2. формирует предсказуемый event/runtime contract для ticks / bars / orderbook;
3. собирает ликвидностные метрики по символам;
4. строит `universe_raw` и `universe_admissible`;
5. вычисляет `universe confidence`;
6. влияет на readiness/degradation платформы через явный orchestration contract;
7. интегрируется в production runtime так же строго, как Risk Engine после `P_5_1`.

## 🔒 ОБЯЗАТЕЛЬНЫЕ ИНЖЕНЕРНЫЕ ПРИНЦИПЫ ФАЗЫ

### 1. Один source of truth для market data
Tick / bar / orderbook / symbol-metrics не должны иметь конкурирующие runtime sources внутри платформы.

### 2. Data quality важнее объёма данных
Лучше честная деградация и явный gap signal, чем молчаливое принятие токсичных данных.

### 3. Universe admissibility — это риск-инфраструктура
`UniverseEngine` — не cosmetic ranking, а gate на то, чем система вообще имеет право торговать.

### 4. Никакого нового bootstrap хаоса
Новый market data runtime должен входить в уже существующий composition-root discipline.

### 5. Event contracts должны быть типизированы и наблюдаемы
Нельзя строить “быстрый” data layer на полусыром transport vocabulary без audit/metrics/readiness semantics.

### 6. Future integrations должны быть заложены, но не фальсифицированы
Если ranked universe или OpportunityEngine ещё не готовы — они должны быть отражены как future contract, а не как будто уже production-ready feature.

## 🧾 DEFINITION OF DONE

Фаза `P_6` считается завершённой только если одновременно выполнено всё ниже:

1. Платформа получила production-ready `Market Data Layer` в текущей архитектуре проекта.
2. Tick / bar / orderbook / universe contracts реализованы и типизированы.
3. `UniverseEngine` реально определяет admissible trading universe, а не является декоративным сервисом.
4. Data quality issues не скрываются, а переводятся в явные runtime/operator signals.
5. Market data runtime встроен в health/readiness/degradation semantics платформы.
6. Integration tests подтверждают реальный runtime wiring нового слоя.
7. Документация и README отражают фактическое состояние `P_6`, а не только замысел prompt.

Для closure `P_6` это трактуется так:
- admissible universe определяется не декоративно, а реальным runtime/policy path;
- конкретная реализация может быть выражена через `MarketDataRuntime` и `UniversePolicy`, если именно они являются фактическим production-compatible кодом фазы.
