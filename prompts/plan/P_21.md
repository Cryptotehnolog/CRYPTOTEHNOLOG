# P_21 - Reporting Artifact Foundation

**Статус:** Phase opening truth assembled  
**Дата:** 2026-03-24

## 1. Смысл фазы

`P_21` открывает `Reporting Artifact Foundation` как узкую artifact-first reporting line.

Эта линия нужна не для построения широкой analytics/reporting platform, а для
формализации typed reporting artifacts поверх уже существующих typed truths,
которые были собраны в:

- `Validation Foundation`;
- `Paper Trading Foundation`;
- `Backtesting / Replay Foundation`.

`P_21` на opening-step не является:

- dashboard / operator line;
- analytics runtime/platform;
- comparison / ranking platform;
- optimization / research line;
- historical data platform.

## 2. Стартовая truth после `v1.20.0`

После `v1.20.0` и post-`P_20` hardening проект уже имеет:

- `ValidationReviewCandidate` как narrow review truth;
- `PaperRehearsalCandidate` как narrow rehearsal truth;
- `ReplayCandidate` как narrow replay truth;
- жёстко зафиксированные ADR boundaries, которые выносят analytics/reporting
  за пределы `Validation`, `Paper` и `Replay`.

Это делает возможным открыть не broad analytics line, а более узкий и честный
future contour:

- `Reporting Artifact Foundation`.

Opening truth этой фазы:

- `P_21 = Reporting Artifact Foundation`
- это узкая artifact-first reporting line;
- она read-only потребляет existing typed candidate truths;
- она владеет только typed reporting contracts, report artifacts, bundle truth,
  deterministic artifact assembly и provenance/read-only discipline.

## 3. In Scope

В scope `P_21` на opening-step входят только:

- отдельный package boundary для reporting artifact layer;
- typed reporting contracts;
- `ValidationReportArtifact`;
- `PaperReportArtifact`;
- `ReplayReportArtifact`;
- `ReportingArtifactBundle`;
- deterministic artifact assembly поверх existing typed truths;
- provenance/read-only discipline;
- narrow query/report-first artifact surface, если он нужен для artifact retrieval,
  но без runtime/service orchestration.

## 4. Exact Upstream Inputs

`P_21` read-only потребляет только:

- `ValidationReviewCandidate`;
- `PaperRehearsalCandidate`;
- `ReplayCandidate`.

Adjacent metrics truth может позже использоваться только как secondary input,
если это потребуется для artifact metadata, но не как ядро opening-scope линии.

## 5. Exact Ownership Boundary

`Reporting Artifact Foundation` владеет только:

- typed reporting contracts;
- artifact types и их invariants;
- artifact bundle truth;
- provenance metadata;
- deterministic derived-report assembly.

`Reporting Artifact Foundation` не владеет:

- lifecycle source truths upstream layers;
- runtime orchestration upstream layers;
- decision semantics upstream layers;
- presentation/UI semantics;
- delivery semantics;
- comparison/benchmark semantics.

Это означает:

- `Validation` остаётся owner-ом review truth;
- `Paper` остаётся owner-ом rehearsal truth;
- `Replay` остаётся owner-ом replay truth;
- `Reporting` только формирует derived reporting artifacts поверх already existing truths.

## 6. Provenance / Read-Only Discipline

Каждый report artifact должен быть:

- derived, а не source-of-truth;
- связан с конкретным upstream candidate;
- способен явно показать source layer;
- обязан сохранять provenance/reference truth;
- не должен перезаписывать upstream status/decision/ownership semantics.

Минимальная honest provenance discipline для opening-step:

- `artifact_id`;
- `artifact_kind`;
- `source_candidate_id`;
- `source_layer`;
- `generated_at`;
- `symbol`;
- `exchange`;
- `timeframe`, если применимо;
- `status`;
- `summary`;
- `reason_code` или equivalent source carry-over;
- optional `score` / `confidence`, только если они уже есть в upstream truth;
- `metadata`;
- explicit provenance/reference fields.

## 7. Почему package boundary нужна уже на opening

Package boundary нужна уже на opening-step, потому что без неё future reporting
truth начнёт сразу расползаться по:

- `validation`;
- `paper`;
- `backtest`;
- future dashboard / operator surface.

Отдельный package boundary нужен как ownership lock:

- reporting layer читает существующие truths;
- reporting layer не забирает ownership у upstream phases;
- reporting line не маскируется под расширение уже закрытых фаз.

## 8. Почему runtime/service boundary пока не нужна

Runtime/service boundary на opening-step пока не нужна, потому что она слишком
рано потянет за собой:

- orchestration;
- persistence/service semantics;
- dashboard coupling;
- notification/delivery behavior;
- broader analytics/runtime platform expectations.

Opening-scope линии честнее удерживается как:

- contracts + artifacts + deterministic assembly,

а не как:

- analytics runtime;
- reporting service;
- dashboard-backed platform.

## 9. Out Of Scope

На opening-step жёстко вне scope:

- dashboard/UI;
- operator workflows;
- comparison/ranking;
- optimization / Monte Carlo / walk-forward;
- historical data platform ownership;
- plotting;
- research lab semantics;
- analytics runtime/platform;
- notification/delivery surface;
- `Execution` ownership;
- `OMS` ownership;
- `Manager` ownership;
- takeover `Validation`;
- takeover `Paper`;
- takeover `Replay`.

## 10. Acceptance Criteria / Definition of Done

Фаза может считаться closure-ready только если:

- существует отдельный reporting package boundary;
- typed reporting contracts формально определены;
- `ValidationReportArtifact`, `PaperReportArtifact`, `ReplayReportArtifact`
  и `ReportingArtifactBundle` существуют как explicit typed truth;
- upstream inputs остаются strictly read-only;
- provenance/read-only discipline enforced явно и проверяемо;
- opening implementation не вводит runtime/service platform;
- opening implementation не создаёт scope inflation в dashboard/comparison/optimization lines;
- README, plan, result doc и final ADR не противоречат этой узкой opening truth.

## 11. Anti-Scope Guards

В рамках `P_21` нельзя честно делать вид, что artifact-first reporting line уже
является:

- analytics platform;
- reporting delivery platform;
- dashboard/operator surface;
- comparison/ranking hub;
- benchmarking/optimization line;
- historical data platform.

Любой шаг, который реально требует этих semantics, должен открываться отдельно,
а не маскироваться под normal continuation `P_21`.

## 12. Note On Historical / Reference Materials

Historical broad analytics/reporting expectations используются только как
reference-only context.

Implementation truth для `P_21` задаётся только:

- current plan truth;
- current ADR boundaries;
- current README truth;
- current code truth существующих upstream layers.

## 13. Note On ADR Before Finalization

На opening-step отдельный ADR для `P_21` пока не обязателен.

Честная последовательность для этой линии:

1. сначала открыть `P_21` как узкую `Reporting Artifact Foundation`;
2. затем, если line реально идёт к finalization, зафиксировать отдельный ADR
   до formal finalization;
3. не позволять interpretation drift расширить линию в dashboard/runtime/comparison semantics.
