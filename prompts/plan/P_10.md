## 🧱 АРХИТЕКТУРНЫЕ ГРАНИЦЫ

### 1. Execution layer является consumer-ом strategy truth

Execution layer:

- потребляет strategy action candidate truth;
- не владеет `market_data`, `analysis`, `intelligence`, `signals`, `strategy` или `risk` truth;
- не подменяет portfolio governance;
- не подменяет future OMS / advanced execution line.

### 2. Bootstrap не владеет execution semantics

Composition root:

- wiring-ит existing runtime contours;
- передаёт execution layer только existing truths;
- не собирает execution contracts вместо execution layer;
- не реализует execution decision logic.

### 3. Event truth не подменяет state/query truth

Execution layer может публиковать typed events, но основная truth должна оставаться:

- query/state-first;
- operator-visible;
- deterministic.

### 4. Runtime-time truth живёт в runtime

Expiry / invalidation / degraded semantics определяются в execution runtime, а не маскируются под dataclass-level temporal truth.

### 5. Execution layer не уводится в широкую external-connectivity platform

Первый execution contour не требует сразу:

- multi-exchange connectivity;
- advanced exchange adapters;
- full order router;
- complex fill orchestration.

Foundation сначала фиксирует ownership, contracts и runtime boundary.

## 🧩 RUNTIME / BOUNDARY DISCIPLINE

`P_10` строится по тем же дисциплинам, что уже закреплены в `P_7-P_9`:

- explicit runtime entrypoint;
- no hidden bootstrap;
- no import-time subscriptions;
- diagnostics sink как operator-visible truth;
- narrow ingest boundary;
- честное разделение structural invariants и runtime-time semantics.

Итоговый boundary фазы:

- execution runtime получает strategy action candidate truth через явный ingest path;
- execution layer сама внутри собирает execution context;
- bootstrap only wires strategy truths into runtime;
- query/state surface позволяет честно наблюдать execution state без необходимости читать только events.
